ความรู้พื้นฐานเกี่ยวกับฐานข้อมูล

ภาพรวมของหน่วย — ปูพื้นฐานเรื่องข้อมูลและฐานข้อมูล แบ่งเป็น 3 ตอน:
• ตอน 1.1 แนวคิดเกี่ยวกับข้อมูลและสารสนเทศ
• ตอน 1.2 ความรู้พื้นฐานเกี่ยวกับระบบฐานข้อมูลและ DBMS
• ตอน 1.3 การประยุกต์ใช้งานฐานข้อมูล

ตอนที่ 1.1 ข้อมูลและสารสนเทศ

ข้อมูล (Data) = ข้อเท็จจริง/เรื่องราวที่ยังไม่ผ่านการประมวลผล เช่น ตัวเลข ตัวอักขระ เสียง ภาพ  ·  สารสนเทศ (Information) = ผลที่ได้จากการนำข้อมูลมาประมวลผลจนนำไปใช้ประโยชน์และสนับสนุนการตัดสินใจได้

คุณสมบัติของข้อมูล/สารสนเทศที่ดี: ถูกต้อง · เป็นปัจจุบัน/ทันเวลา · สมบูรณ์ · กระชับชัดเจน · ตรงความต้องการผู้ใช้ · ต่อเนื่อง

กระบวนการประมวลผลข้อมูล: การนำเข้า (Input) → การประมวลผล (Process) → การส่งออกผล (Output) → ผลตอบกลับ (Feedback) — Feedback ช่วยให้ผู้ใช้ทราบข้อผิดพลาดที่ต้องแก้ไข

ลำดับชั้นการจัดเก็บข้อมูล (เล็ก → ใหญ่)

บิต (bit) → ไบต์ (byte) → เขตข้อมูล (field) → ระเบียน (record) → แฟ้มข้อมูล (file) → ฐานข้อมูล (database) — เช่น ข้อมูลพนักงาน 1 คน = 1 ระเบียน, แต่ละช่อง = เขตข้อมูล

แฟ้มข้อมูล (File) — โครงสร้าง 4 แบบ และข้อจำกัด

โครงสร้าง: แบบเรียง (serial) · แบบเรียงลำดับ (sequential) · แบบสุ่ม/ตรง (random/direct) · แบบเรียงลำดับดัชนี (indexed sequential — ยืดหยุ่นในการเข้าถึงมากที่สุด)

ข้อจำกัดของแฟ้มข้อมูลแบบเดิม: ข้อมูลซ้ำซ้อน · ขึ้นต่อกันกับโปรแกรม · ขาดความยืดหยุ่น · ขาดความปลอดภัย · ใช้ร่วมกันไม่สะดวก · ไม่มีศูนย์กลางควบคุม

ตอนที่ 1.2 ระบบฐานข้อมูลและ DBMS

ระบบฐานข้อมูล = ระบบที่รวมข้อมูลที่สัมพันธ์กันไว้ที่เดียว เพื่อเข้าถึง/เรียกใช้/ประมวลผล/ดูแลรักษาได้อย่างมีประสิทธิภาพ โดยอาศัยซอฟต์แวร์ DBMS — ลักษณะที่พึงประสงค์: ไม่ซ้ำซ้อน · ถูกต้องไม่ขัดแย้ง · ปลอดภัย · เป็นอิสระจากโปรแกรม (data independence)

องค์ประกอบ 5 ส่วน: ฮาร์ดแวร์ · ซอฟต์แวร์ · ข้อมูล · บุคลากร · กระบวนการ/ขั้นตอนการทำงาน  (⚠ งบประมาณ ไม่ใช่ องค์ประกอบ)

DBMS = ซอฟต์แวร์ตัวกลางระหว่างผู้ใช้กับฐานข้อมูล บริหารฐานข้อมูลให้มีประสิทธิภาพ (รวมถึงจัดทำพจนานุกรมข้อมูล) — เลือกจาก ต้นทุนเทียบประโยชน์/คุณสมบัติ/โครงสร้าง เช่น PostgreSQL, MySQL, MS SQL Server · ภาษาเรียกค้น = ภาษาคิวรี (Query Language) เช่น SQL

ตอนที่ 1.3 การประยุกต์ใช้งานฐานข้อมูล

ด้านตัวอย่างข้อมูลที่จัดเก็บ
การศึกษาระบบทะเบียนนักศึกษา (นักศึกษา ชุดวิชา อาจารย์ ห้องเรียน การลงทะเบียน ผลการเรียน), ระบบห้องสมุด (หนังสือ สมาชิก การยืม-คืน)
ธุรกิจระบบขายออนไลน์ (สินค้า ลูกค้า รายการสั่งซื้อ การชำระเงิน การจัดส่ง)
วิทยาศาสตร์/การแพทย์ระบบผู้ป่วยในโรงพยาบาล (ประวัติผู้ป่วย การตรวจรักษา ยา/เวชภัณฑ์ แพทย์)
แนวตอบกิจกรรมระหว่างเรียน
1.1.1 ความแตกต่างของข้อมูลกับสารสนเทศ / คุณสมบัติที่ดี
ข้อมูล = ข้อเท็จจริงที่ยังไม่ประมวลผล; สารสนเทศ = ข้อมูลที่ประมวลผลแล้วใช้สนับสนุนการตัดสินใจได้ คุณสมบัติที่ดี: ถูกต้อง เป็นปัจจุบัน สมบูรณ์ กระชับ ตรงความต้องการ ต่อเนื่อง
1.1.2 หลักบริหารจัดการข้อมูล / ลำดับชั้นการจัดเก็บ
หลักบริหาร: เข้าถึงข้อมูลได้ ปลอดภัย แก้ไขได้ในอนาคต; ลำดับชั้น: บิต → ไบต์ → เขตข้อมูล → ระเบียน → แฟ้มข้อมูล → ฐานข้อมูล
1.1.3 ปัญหาแฟ้มข้อมูลแบบเดิม / โครงสร้างแฟ้มข้อมูล
ปัญหา: ซ้ำซ้อน ขึ้นต่อกัน ขาดความยืดหยุ่น/ปลอดภัย/แบ่งปัน; โครงสร้าง 4 แบบ: เรียง, เรียงลำดับ, สุ่ม/ตรง, เรียงลำดับดัชนี
1.2.1 ความหมายระบบฐานข้อมูล / ลักษณะที่พึงประสงค์
ระบบที่รวมข้อมูลสัมพันธ์กันไว้ที่เดียว เข้าถึง-ประมวลผล-ดูแลได้อย่างมีประสิทธิภาพ; พึงประสงค์: ไม่ซ้ำซ้อน ถูกต้องไม่ขัดแย้ง ปลอดภัย เป็นอิสระจากโปรแกรม
1.2.2 ความหมาย DBMS / การเลือกใช้ / ตัวอย่าง
DBMS = ซอฟต์แวร์บริหารฐานข้อมูลให้มีประสิทธิภาพ; เลือกจากต้นทุน-ประโยชน์ คุณสมบัติ โครงสร้าง; ตัวอย่าง: PostgreSQL, MySQL, MS SQL Server
1.3.1 ข้อมูลในระบบทะเบียนนักศึกษา / ระบบห้องสมุด
ทะเบียน: นักศึกษา ชุดวิชา อาจารย์ ห้องเรียน การลงทะเบียน ผลการเรียน; ห้องสมุด: รายละเอียดหนังสือ สมาชิก รายการยืม-คืน
1.3.2 การประยุกต์ฐานข้อมูลด้านธุรกิจ
ระบบขายออนไลน์: สินค้า ลูกค้า รายการสั่งซื้อ การชำระเงิน การจัดส่ง
1.3.3 การประยุกต์ฐานข้อมูลด้านวิทยาศาสตร์/การแพทย์
ระบบผู้ป่วยในโรงพยาบาล: ประวัติผู้ป่วย การตรวจรักษา ยา/เวชภัณฑ์ แพทย์
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดกล่าวถึงข้อมูลและสารสนเทศไม่ถูกต้อง
เฉลย: ข. คุณสมบัติของข้อมูลที่ดีเหมือนกับคุณสมบัติของสารสนเทศที่ดี — คุณสมบัติข้อมูลที่ดีกับสารสนเทศที่ดีไม่เหมือนกันทั้งหมด ข้อ ข จึงผิด
ข้อ 2. ชื่อของนักศึกษา จัดเป็นข้อมูลชนิดใด
เฉลย: ค. ตัวอักขระ — ชื่อเป็นตัวอักษร/อักขระ ไม่ใช่ตัวเลข เสียง หรือภาพ
ข้อ 3. ข้อใดจัดเป็นข้อมูล (data)
เฉลย: จ. ถูกทั้ง ก. และ ค. — รหัสนักศึกษาและลักษณะของปลาหางนกยูงเป็นข้อเท็จจริงดิบ = ข้อมูล; จำนวน/อายุสูงสุด เป็นผลประมวลผล = สารสนเทศ
ข้อ 4. ข้อใดไม่จัดเป็นคุณสมบัติของข้อมูลที่ดี
เฉลย: ก. มีจำนวนมาก — ข้อมูลที่ดีวัดที่คุณภาพ ไม่ใช่ปริมาณมาก
ข้อ 5. กระบวนการประมวลผลข้อมูลประกอบด้วยส่วนสำคัญ ยกเว้นข้อใด
เฉลย: ข. การตรวจสอบข้อมูล — องค์ประกอบหลัก = นำเข้า–ประมวลผล–ส่งออก–ผลตอบกลับ; การตรวจสอบข้อมูลไม่ใช่ขั้นตอนหลัก
ข้อ 6. ข้อใดไม่จัดเป็นประโยชน์ของการบริหารจัดการข้อมูล
เฉลย: ค. เพื่อให้ข้อมูลมีความถูกต้องทันสมัย — ข้อนี้เป็นคุณสมบัติของข้อมูล ไม่ใช่ประโยชน์ของการบริหารจัดการโดยตรง
ข้อ 7. หน่วยข้อมูลใดมีขนาดใหญ่ที่สุด
เฉลย: ค. ฐานข้อมูล (database) — บิต < ไบต์ < field < record < file < database (ใหญ่สุด)
ข้อ 8. ข้อใดไม่ใช่ข้อจำกัดของการจัดเก็บด้วยแฟ้มข้อมูล
เฉลย: ง. ขาดความสะดวกในการค้นหาข้อมูล — ข้อจำกัดเด่น = ขึ้นต่อกัน ซ้ำซ้อน ใช้ร่วมไม่สะดวก ปลอดภัยยาก
ข้อ 9. ข้อใดไม่ใช่องค์ประกอบของระบบฐานข้อมูล
เฉลย: ข. งบประมาณ — องค์ประกอบ = ฮาร์ดแวร์ ซอฟต์แวร์ ข้อมูล บุคลากร กระบวนการ
ข้อ 10. ระบบทะเบียนนักศึกษา ข้อมูลใดไม่จำเป็นต้องจัดเก็บ
เฉลย: ง. ข้อมูลมหาวิทยาลัย — ข้อมูลมหาวิทยาลัยเป็นค่าคงที่เดียว ไม่จำเป็นต้องเก็บในฐานข้อมูลทะเบียน
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดจัดเป็นข้อมูลเชิงคุณภาพ
เฉลย: ค. รูปร่าง — เชิงคุณภาพบอกคุณลักษณะ เช่น รูปร่าง สี; น้ำหนัก/คะแนน/รายได้ เป็นเชิงปริมาณ
ข้อ 2. ข้อใดกล่าวถึงสารสนเทศไม่ถูกต้อง
เฉลย: ง. คุณสมบัติของสารสนเทศที่ดีเหมือนกับคุณสมบัติของข้อมูลที่ดี — คุณสมบัติสารสนเทศที่ดีไม่เหมือนข้อมูลที่ดีทั้งหมด
ข้อ 3. องค์ประกอบใดที่ทำให้ผู้ใช้ทราบข้อผิดพลาดที่ต้องแก้ไข
เฉลย: ง. ผลตอบกลับ (Feedback) — Feedback แจ้งข้อผิดพลาดเพื่อนำไปปรับปรุง
ข้อ 4. ข้อมูลพนักงาน 1 คน (รหัส ชื่อ ที่อยู่ ตำแหน่ง) ข้อใดถูกต้อง
เฉลย: จ. (ข้อที่รวม ค. และ ง.) — ชื่อเก็บเป็นอักขระ และมี 4 เขตข้อมูล; ก/ข สลับนิยาม field กับ record
ข้อ 5. โครงสร้างแฟ้มข้อมูลแบบใดยืดหยุ่นในการเข้าถึงมากกว่าแบบอื่น
เฉลย: ง. แฟ้มข้อมูลแบบเรียงลำดับดัชนี (indexed sequential) — เข้าถึงได้ทั้งเรียงลำดับและสุ่มผ่านดัชนี จึงยืดหยุ่นสุด
ข้อ 6. ข้อใดคือข้อจำกัดในการเข้าใช้งานข้อมูลแบบแฟ้มข้อมูล
เฉลย: จ. ถูกทุกข้อ — เพิ่มข้อมูลผิดง่าย แก้โปรแกรมกระทบแฟ้มอื่น ข้อมูลขัดแย้ง ไม่มีศูนย์กลางควบคุม
ข้อ 7. ข้อใดคือข้อจำกัดของการจัดเก็บด้วยฐานข้อมูล
เฉลย: ง. ค่าใช้จ่ายที่เกี่ยวข้องในการจัดเก็บด้วยฐานข้อมูล — ข้อจำกัดคือค่าใช้จ่ายสูง (ซอฟต์แวร์/ฮาร์ดแวร์/บุคลากร)

2.1.1 โครงสร้างฐานข้อมูลแบ่งเป็นกี่ระดับ อะไรบ้าง

คำนึงถึง: ข้อจำกัดการตั้งชื่อตาราง/แอตทริบิวต์, ชนิดและขนาดของแอตทริบิวต์, การกำหนดคีย์หลัก/คีย์นอกและความสัมพันธ์ระหว่างตาราง

ความสำคัญ: เป็นแหล่งรวบรวมแนวคิด วัตถุ เหตุการณ์จริงขององค์กรพร้อมความสัมพันธ์ ช่วยให้ผู้ออกแบบกับผู้ใช้เข้าใจตรงกัน และ ไม่ขึ้นกับชนิดของ DBMS

การออกแบบฐานข้อมูล 3 ระดับ: ระดับแนวคิด (Conceptual) → ระดับตรรกะ (Logical, ทำ Normalization) → ระดับกายภาพ (Physical, ปรับเข้ากับ DBMS จริง)

องค์ประกอบหลักของแผนภาพ E-R (4 อย่าง) และสัญลักษณ์

องค์ประกอบสัญลักษณ์ตัวอย่าง
เอนทิตี (Entity)สี่เหลี่ยมผืนผ้าลูกค้า, สินค้า, นักศึกษา
แอตทริบิวต์ (Attribute)วงรี (ellipse)ชื่อ, ราคา, ที่อยู่
คีย์หลัก (Primary Key)วงรี + ขีดเส้นใต้รหัสนักศึกษา
ความสัมพันธ์ (Relationship)สี่เหลี่ยมข้าวหลามตัดซื้อ, เช่า, สังกัด
เอนทิตีอ่อน (Weak Entity)สี่เหลี่ยมสองชั้น—

⚠ คีย์สำรอง/Foreign Key ไม่ใช่ องค์ประกอบหลักของ E-R

ดีกรีและข้อบังคับของความสัมพันธ์

ดีกรี 3 ชนิด: ยูนารี (Unary) · ไบนารี (Binary, 2 เอนทิตี) · เทอร์นารี (Ternary, 3 เอนทิตี) · ข้อบังคับ 2 ชนิด: คาร์ดินาลลิตี (Cardinality) และการมีส่วนร่วม (Participation)

ชนิด (cardinality)ตัวอย่าง
1:1นักศึกษา มี บัตรนักศึกษา (1 คน : 1 ใบ)
1:Mบริษัท มี ตำแหน่งงาน (1 บริษัท : หลายตำแหน่ง)
M:Nนักศึกษา ลงทะเบียน ชุดวิชา (หลายคน : หลายวิชา)

ตอนที่ 3.2 ขั้นตอนการสร้างแผนภาพ E-R

จำแนกข้อมูลเป็น 4 ประเภท: เอนทิตี · แอตทริบิวต์ · คีย์หลัก · ความสัมพันธ์

  1. นำความสัมพันธ์ที่กำหนดไว้มาเปลี่ยนเป็นสัญลักษณ์
  2. กำหนดชนิดความสัมพันธ์ (1:1, 1:M, M:N)
  3. นำแต่ละส่วนมาประกอบเป็นโครงร่าง
  4. ปรับรูปทรงของแผนภาพ E-R ให้เหมาะสม
  5. ใส่แอตทริบิวต์ของเอนทิตีและความสัมพันธ์
แนวตอบกิจกรรมระหว่างเรียน
3.1.1 ความสำคัญของการออกแบบระดับแนวคิด / แผนภาพที่นิยมใช้
เป็นแหล่งรวมแนวคิด วัตถุ เหตุการณ์ + ความสัมพันธ์ขององค์กร ช่วยสื่อสารผู้ออกแบบ-ผู้ใช้ ไม่ขึ้นกับ DBMS; แผนภาพที่นิยม = แผนภาพ E-R
3.1.2 องค์ประกอบหลักของ E-R / ดีกรี / ข้อบังคับ
องค์ประกอบ: เอนทิตี แอตทริบิวต์ คีย์หลัก ความสัมพันธ์; ดีกรี 3 ชนิด (ยูนารี/ไบนารี/เทอร์นารี); ข้อบังคับ 2 ชนิด (คาร์ดินาลลิตี + การมีส่วนร่วม)
3.2.1 ข้อมูลที่ใช้ออกแบบ E-R มีกี่ประเภท
4 ประเภท: เอนทิตี, แอตทริบิวต์, คีย์หลัก, ความสัมพันธ์
3.2.2 ขั้นตอนการสร้างแผนภาพ E-R
5 ขั้นตอน: เปลี่ยนความสัมพันธ์เป็นสัญลักษณ์ → กำหนดชนิดความสัมพันธ์ → ประกอบโครงร่าง → ปรับรูปทรง → ใส่แอตทริบิวต์
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ความสำคัญของการออกแบบระดับแนวคิดต่อการพัฒนาระบบสารสนเทศ
เฉลย: จ. เป็นที่รวบรวมแนวคิด วัตถุ เหตุการณ์ และความสัมพันธ์ของข้อมูลในองค์กร — ครอบคลุมครบทั้งแนวคิด วัตถุ เหตุการณ์ และความสัมพันธ์
ข้อ 2. ข้อใดไม่ใช่องค์ประกอบหลักของแผนภาพ E-R
เฉลย: ก. คีย์สำรอง — องค์ประกอบหลัก = เอนทิตี/แอตทริบิวต์/ความสัมพันธ์/เส้นเชื่อม
ข้อ 3. ความสัมพันธ์ใดเป็นแบบ 1:1
เฉลย: ข. นักศึกษามีบัตรนักศึกษา — นักศึกษา 1 คน มีบัตร 1 ใบ = 1:1
ข้อ 4. ขั้นตอนแรกของการสร้างแผนภาพ E-R
เฉลย: จ. วิเคราะห์ข้อมูลได้ 4 ประเภท (เอนทิตี แอตทริบิวต์ ความสัมพันธ์ คีย์หลัก) — เริ่มจากจำแนกข้อมูลให้ครบ 4 ประเภทก่อน
ข้อ 5. สัญลักษณ์ที่ใช้กับข้อมูล "ลูกค้า"
เฉลย: ก. สี่เหลี่ยมผืนผ้า — ลูกค้า = เอนทิตี ใช้สี่เหลี่ยมผืนผ้า
ข้อ 6. สัญลักษณ์ที่แสดง "ความสัมพันธ์"
เฉลย: ค. สี่เหลี่ยมข้าวหลามตัด (diamond) — ความสัมพันธ์ใช้สัญลักษณ์ข้าวหลามตัด
ข้อ 7. (แผนภาพ) รถยนต์ — มี — ยี่ห้อ เป็นความสัมพันธ์แบบใด
เฉลย: ข. 1:M — ยี่ห้อหนึ่งมีรถยนต์ได้หลายคัน = 1:M
ข้อ 8. (แผนภาพ) พนักงาน M—สังกัด→ฝ่าย ลูกศรชี้ตำแหน่งใด
เฉลย: ก. 1 — พนักงานหลายคนสังกัดหนึ่งฝ่าย ลูกศรชี้ด้าน "ฝ่าย" = ด้าน 1
ข้อ 9. (สมาชิก-ซื้อ-สินค้า) ข้อใดทำให้สมบูรณ์ที่สุด
เฉลย: ง. จำนวนที่ซื้อ — วงรีว่างที่ความสัมพันธ์ "ซื้อ" ควรเป็นแอตทริบิวต์ "จำนวนที่ซื้อ"
ข้อ 10. (ก–จ) ข้อใดปรับรูปทรงแผนภาพ E-R ได้ถูกต้องที่สุด
เฉลย: ง. — ตัวเลือก ง วางเอนทิตี/ความสัมพันธ์/แอตทริบิวต์ครบถูกตำแหน่งและไม่มีวงรีว่าง/ซ้ำ
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ความสำคัญของการออกแบบระดับแนวคิด
เฉลย: ง. เป็นที่รวบรวมวัตถุ เหตุการณ์จริงขององค์กร และความสัมพันธ์ของข้อมูล — ครอบคลุมแนวคิด/วัตถุ/เหตุการณ์/ความสัมพันธ์มากที่สุด
ข้อ 2. ข้อใดไม่ใช่องค์ประกอบหลักของแผนภาพ E-R
เฉลย: จ. Foreign Key (คีย์นอก) — FK เกิดตอนแปลงเป็นตาราง ไม่ใช่องค์ประกอบของ E-R
ข้อ 3. ความสัมพันธ์ใดเป็นแบบ M:1
เฉลย: ค. พนักงานเป็นหัวหน้าคนงาน — พนักงานหลายคนมีหัวหน้าหนึ่งคน = M:1
ข้อ 4. ขั้นตอนแรกของการสร้างแผนภาพ E-R
เฉลย: จ. จำแนกประเภทข้อมูลเป็น Attribute, Entity, Relationship, Primary Key — จำแนกข้อมูลให้ครบ 4 ประเภทก่อนเป็นขั้นแรก
ข้อ 5. สัญลักษณ์ของ Attribute
เฉลย: ง. วงรี (ellipse) — แอตทริบิวต์ใช้สัญลักษณ์วงรี
ข้อ 6. สัญลักษณ์ของ Relationship
เฉลย: ค. สี่เหลี่ยมข้าวหลามตัด — ความสัมพันธ์ใช้ข้าวหลามตัด
ข้อ 7. (แผนภาพ) สมาชิก — มี — บัตรสมาชิก เป็นความสัมพันธ์แบบใด
เฉลย: ก. 1:1 — สมาชิก 1 คน มีบัตร 1 ใบ = 1:1
ข้อ 8. (แผนภาพ) พนักงาน←ทำ—M บัตรสมาชิก ลูกศรชี้ตำแหน่งใด
เฉลย: ง. 1 — พนักงานหนึ่งคนทำบัตรได้หลายใบ ลูกศรชี้ด้าน "พนักงาน" = ด้าน 1
ข้อ 9. (สมาชิก-ชำระเงิน-พนักงาน) ข้อใดทำให้สมบูรณ์
เฉลย: ค. จำนวนที่ขาย — เติมวงรีว่างด้วยแอตทริบิวต์ที่เหมาะกับบริบท
ข้อ 10. (ก–จ) ข้อใดปรับรูปทรงแผนภาพ E-R ได้ถูกต้องที่สุด
เฉลย: ก. — ตัวเลือก ก จัดวางองค์ประกอบครบถูกต้องและไม่มีวงรีว่าง/ซ้ำ
สรุปประเด็นที่ออกสอบบ่อย
  • องค์ประกอบ E-R: Entity / Attribute / Primary Key / Relationship (ไม่ใช่ FK, คีย์สำรอง)
  • สัญลักษณ์: เอนทิตี=สี่เหลี่ยม, แอตทริบิวต์=วงรี, ความสัมพันธ์=ข้าวหลามตัด, คีย์หลัก=วงรีขีดเส้นใต้
  • เอนทิตีอ่อน: สี่เหลี่ยมสองชั้น
  • ดีกรี: ยูนารี / ไบนารี / เทอร์นารี  ·  ข้อบังคับ: คาร์ดินาลลิตี + การมีส่วนร่วม
  • ชนิดความสัมพันธ์: 1:1 (มีบัตร), 1:M (บริษัท-ตำแหน่ง), M:N (ลงทะเบียน)
  • 5 ขั้นตอน: สัญลักษณ์→ชนิดความสัมพันธ์→โครงร่าง→ปรับรูปทรง→ใส่แอตทริบิวต์
  • ระดับการออกแบบ: แนวคิด → ตรรกะ(normalize) → กายภาพ

การออกแบบฐานข้อมูลระดับตรรกะ

ภาพรวมของหน่วย — การออกแบบระดับตรรกะ (Logical Design): แปลงแผนภาพ E-R ให้กลายเป็นตาราง (สคีมา) ที่ซ้ำซ้อนน้อยที่สุด แบ่งเป็น 2 ตอน:
• ตอน 4.1 แนวคิดและขั้นตอนการแปลง E-R เป็นตาราง
• ตอน 4.2 ตัวอย่างการประยุกต์และข้อพิจารณา

ตอนที่ 4.1 แนวคิดและขั้นตอนการแปลง E-R เป็นตาราง

การออกแบบระดับตรรกะ = แปลงแผนภาพ E-R ให้อยู่ในรูปตารางที่ซ้ำซ้อนน้อยที่สุด มี 2 ขั้นตอนใหญ่: (1) แปลงเอนทิตีเป็นตาราง (2) แปลงความสัมพันธ์เป็นตาราง

ทำไมต้องเรียนแปลงด้วยมือ? เพราะการแก้ปัญหามีมากกว่า 1 วิธีตามสถานการณ์ และบางกรณีเครื่องมือ CASE ทำไม่ได้ (เช่น เทอร์นารี/ซับไทป์-ซูเปอร์ไทป์)

กฎการแปลง E-R เป็นตาราง (หัวใจของหน่วย)

กรณีวิธีแปลง
เอนทิตีปกติ (Strong)สร้างเป็น 1 ตาราง คีย์หลักของเอนทิตี = PK ของตาราง
เอนทิตีอ่อน (Weak)สร้างตาราง โดยนำ PK ของเอนทิตีเจ้าของมารวมเป็นส่วนหนึ่งของคีย์
ความสัมพันธ์ 1:1ไม่สร้างตารางใหม่ นำ PK ฝั่งหนึ่งไปเป็น FK อีกฝั่ง
ความสัมพันธ์ 1:Mไม่สร้างตารางใหม่ นำ PK ฝั่ง "1" ไปเป็น FK ในตารางฝั่ง "M"
ความสัมพันธ์ M:Nต้องสร้างตารางใหม่ นำ PK ทั้งสองฝั่งมารวมเป็นคีย์ + ใส่แอตทริบิวต์ของความสัมพันธ์
แอตทริบิวต์พหุค่าแยกเป็นตารางใหม่ต่างหาก

จุดสำคัญ: M:N เท่านั้นที่ต้องสร้าง "ตารางใหม่" และตารางนั้นช่วยลดความซ้ำซ้อน

4 ขั้นตอนการแปลง "ความสัมพันธ์" เป็นตาราง

  1. นำชื่อความสัมพันธ์มาสร้างเป็นชื่อตาราง
  2. นำคีย์หลักของเอนทิตีที่เกี่ยวข้องมาใส่
  3. นำแอตทริบิวต์ของความสัมพันธ์มาใส่
  4. เลือกคีย์หลักของตาราง

ตัวอย่างการเขียนสคีมา (ขีดเส้นใต้=PK, * = FK)

PRODUCT (product_no, product_des, comp_id*)
APPLY (ap_id*, pos_id*, apply_date)

ตอนที่ 4.2 ข้อพิจารณาและตัวอย่าง

การออกแบบระดับตรรกะลดความซ้ำซ้อนจากการเก็บแบบไฟล์ ถ้าปรับรูปทรง E-R ครบถ้วนจะช่วย (1) ลดจำนวนตาราง (2) ลดความซ้ำซ้อนของแอตทริบิวต์บนความสัมพันธ์ (3) ลดการรวมตารางในขั้นถัดไป — ขั้นถัดจากระดับตรรกะคือ การนอร์มัลไลเซชัน (หน่วย 5)

แนวตอบกิจกรรมระหว่างเรียน
4.1.1 เหตุผลที่ต้องศึกษาการแปลง E-R เป็นตาราง
2 ประเด็น: (1) เรียนรู้วิธีแก้ปัญหาเพราะมีหลายวิธีตามสถานการณ์ (2) บางกรณีเครื่องมือ CASE ทำไม่ได้ เช่น เทอร์นารี/ซับไทป์-ซูเปอร์ไทป์
4.1.2 ขั้นตอนการแปลงความสัมพันธ์เป็นตาราง
แปลงตามชนิด: 1:1, 1:M, M:N, ยูนารี/รีเคอร์ซีฟ, เทอร์นารี โดย M:N ต้องสร้างตารางใหม่
4.2.2 จำแนกประเภทข้อมูล (เอนทิตี/แอตทริบิวต์/คีย์หลัก)
เช่น ลูกค้า (PK=หมายเลขประชาชน), คอมพิวเตอร์ (PK=เลขเครื่อง), พนักงาน (PK=รหัสพนักงาน), บัตรสมาชิก (PK=เลขที่บัตร) พร้อมระบุความสัมพันธ์ระหว่างกัน
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. จากตารางลูกค้า (เลขประจำตัว, ชื่อ, ที่อยู่, เบอร์โทร) ข้อใดแทน primary key ได้
เฉลย: ง. (ตัวเลือกที่มี เลขประจำตัว รวมอยู่) — คีย์หลักต้องไม่ซ้ำ — มีเพียง "เลขประจำตัว" ที่ไม่ซ้ำจึงใช้เป็นคีย์ได้
ข้อ 2. ข้อใดไม่ถูกต้อง (เกี่ยวกับการเลือกคีย์)
เฉลย: ก. ประสิทธิภาพการค้นหาจะเร็วขึ้นถ้าใช้ Single key — ความเร็วค้นหาไม่ได้ขึ้นกับการเป็น single key เสมอไป
ข้อ 3. ข้อมูลสมาชิกที่จัดเก็บในฐานข้อมูลเป็นข้อมูลในข้อใด
เฉลย: ค. เอนทิตีอินสแตนซ์ (entity instance) — ข้อมูลสมาชิกแต่ละราย = อินสแตนซ์หนึ่งของเอนทิตี
ข้อ 4. "พนักงานสังกัดฝ่าย" สอดคล้องกับความสัมพันธ์แบบใด
เฉลย: ข. ความสัมพันธ์แบบไบนารี — เกี่ยวข้อง 2 เอนทิตี (พนักงาน, ฝ่าย) = ไบนารี
ข้อ 5. ขั้นตอนแรกของการสร้างตารางคือข้อใด
เฉลย: ค. การแปลง Entity และ Relationship เป็นตาราง — ระดับตรรกะคือแปลงทั้งเอนทิตีและความสัมพันธ์ให้เป็นตาราง
ข้อ 6. เรียงลำดับขั้นตอนการแปลงความสัมพันธ์เป็นตาราง
เฉลย: ข. 3 4 2 1 — ตั้งชื่อตาราง → ใส่คีย์หลักเอนทิตีที่เกี่ยวข้อง → ใส่แอตทริบิวต์ของความสัมพันธ์ → เลือกคีย์หลัก
ข้อ 7. (แผนภาพ) ชื่อที่ดีที่สุดของ Relationship "ส่ง"
เฉลย: จ. การส่งหนังสือ หรือใบส่งสินค้า — ชื่อความสัมพันธ์ควรสื่อความหมายชัดเจนตามบริบท
ข้อ 8. (แผนภาพ) ตารางที่ได้จากการแปลง Relationship "ส่ง"
เฉลย: ค. — ตารางจากความสัมพันธ์ต้องมีคีย์หลักของเอนทิตีที่เกี่ยวข้อง + แอตทริบิวต์ของความสัมพันธ์
ข้อ 9. (แผนภาพ) ตารางที่ได้จากการแปลง Entity "หนังสือ"
เฉลย: ค. หนังสือ (รหัสหนังสือ, ชื่อหนังสือ, ราคาเช่า, วันที่ซื้อมา) — ตารางจากเอนทิตีมีเฉพาะแอตทริบิวต์ของเอนทิตีนั้น ไม่ต้องมี FK (M:N แยกตารางต่างหาก)
ข้อ 10. การลดความซ้ำซ้อนของข้อมูลเกิดในข้อใด
เฉลย: ง. สร้างตารางจาก Entity ที่มีความสัมพันธ์แบบ 1:M — การจัดการ 1:M ด้วย FK ช่วยจัดระเบียบและลดความซ้ำซ้อน
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดไม่ใช่สาเหตุที่ต้องทราบกระบวนการแปลง E-R เป็นตาราง
เฉลย: ง. ความสัมพันธ์แบบไบนารีจำเป็นต้องแปลงด้วยมือ — ไบนารีเครื่องมือ CASE ทำได้ ที่ทำมือคือกรณีซับซ้อนเช่นเทอร์นารี
ข้อ 2. ข้อใดไม่ถูกต้อง
เฉลย: จ. ประสิทธิภาพการค้นหาจะเร็วขึ้นถ้าใช้ composite key — composite key ไม่ได้ทำให้ค้นหาเร็วขึ้นเสมอไป
ข้อ 3. ถ้าเลือกเก็บข้อมูล "หนังสือ" จะกำหนดให้หนังสือเป็นข้อมูลในข้อใด
เฉลย: ก. Strong Entity — หนังสือมีคีย์หลักของตัวเอง ดำรงอยู่อิสระ = Strong Entity
ข้อ 4. ลูกค้ามีบัตร ATM ความสัมพันธ์สอดคล้องกับข้อใด
เฉลย: ข. Binary Relationship — เกี่ยวข้อง 2 เอนทิตี (ลูกค้า, บัตร ATM) = ไบนารี
ข้อ 5. ขั้นตอนแรกของการสร้างตาราง
เฉลย: ค. การแปลง Entity และ Relationship เป็นตาราง — ระดับตรรกะคือแปลงทั้งเอนทิตีและความสัมพันธ์เป็นตาราง
ข้อ 6. เรียงลำดับขั้นตอนการแปลงความสัมพันธ์เป็นตาราง
เฉลย: ก. 2 4 3 1 — ตั้งชื่อตาราง → ใส่คีย์หลักเอนทิตีที่เกี่ยวข้อง → ใส่แอตทริบิวต์ → เลือกคีย์หลัก
ข้อ 7. (แผนภาพ) ชื่อที่ดีที่สุดของ Relationship "ซื้อ"
เฉลย: ง. การซื้อโปรแกรมสุขภาพ — ชื่อความสัมพันธ์ควรสื่อความหมายชัดตามบริบท
ข้อ 8. (แผนภาพ) ตารางที่ได้จากการแปลง Relationship "ซื้อ"
เฉลย: จ. — ตารางจาก M:N ต้องมีคีย์หลักของเอนทิตีที่เกี่ยวข้องครบ + แอตทริบิวต์ของความสัมพันธ์
ข้อ 9. (แผนภาพ) ตารางที่ได้จากการแปลง Entity "บัตรสมาชิก"
เฉลย: ก. บัตรสมาชิก (เลขที่, ประเภท, ชื่อลูกค้า, วันที่ทำ, วันหมดอายุ, รหัสพนักงาน) — ตารางบัตรสมาชิกมีแอตทริบิวต์ของตน + FK ที่เชื่อมจากความสัมพันธ์ 1:1/1:M
ข้อ 10. การลดความซ้ำซ้อนของข้อมูลเกิดในข้อใด
เฉลย: ข. สร้างตารางจาก Entity ที่มีความสัมพันธ์แบบ 1:M — การจัดการ 1:M ด้วย FK ช่วยลดการเก็บข้อมูลซ้ำ
สรุปประเด็นที่ออกสอบบ่อย
  • ระดับตรรกะ: แปลง E-R → ตารางที่ซ้ำซ้อนน้อยสุด
  • 2 ขั้นตอนใหญ่: แปลงเอนทิตีเป็นตาราง + แปลงความสัมพันธ์เป็นตาราง
  • 1:1 และ 1:M: ไม่สร้างตารางใหม่ ใช้ FK
  • M:N: ต้องสร้างตารางใหม่ (PK ทั้งสองฝั่ง + แอตทริบิวต์)
  • คีย์หลัก ต้องไม่ซ้ำ (เลขประจำตัว/รหัส ใช้ได้)
  • ตารางจากเอนทิตี มีเฉพาะแอตทริบิวต์ของเอนทิตีนั้น
  • ขั้นถัดไป: Normalization (หน่วย 5)
5

การออกแบบโดยวิธีนอร์มัลไลเซชัน

ภาพรวมของหน่วย — การนอร์มัลไลเซชัน (Normalization): จัดการรีเลชันที่ซ้ำซ้อนให้อยู่ในรูปแบบบรรทัดฐาน (Normal Form) แบ่งเป็น 2 ตอน:
• ตอน 5.1 แนวคิด (ปัญหาความซ้ำซ้อน, การขึ้นต่อกันเชิงฟังก์ชัน FD)
• ตอน 5.2 วิธีการนอร์มัลไลเซชัน (1NF → 5NF)

ตอนที่ 5.1 แนวคิดการนอร์มัลไลเซชัน

นอร์มัลไลเซชัน = ทฤษฎีออกแบบฐานข้อมูล จัดการรีเลชันที่ซ้ำซ้อนให้อยู่ในรูปแบบบรรทัดฐาน · วัตถุประสงค์: ลดความซ้ำซ้อนของข้อมูล และรักษาความถูกต้องให้มากที่สุด

ปัญหาจากความซ้ำซ้อน (Anomalies) 3 อย่าง: การเพิ่ม (Insert) · การแก้ไข (Update) · การลบ (Delete)  (⚠ "การสืบค้นข้อมูล" ไม่ใช่ปัญหาจากความซ้ำซ้อน)

การขึ้นต่อกันเชิงฟังก์ชัน (Functional Dependency: FD)

ถ้าแอตทริบิวต์ A ระบุค่าของ B ได้ในทูเพิลเดียวกัน เขียนว่า A → B โดย A = ดีเทอร์มิแนนต์ (determinant), B = ดีเพนเดนซี (dependency)

ชนิด FDความหมาย
เต็มค่า (Full)ฝั่งขวาขึ้นกับทุกแอตทริบิวต์ฝั่งซ้ายของคีย์ (ทั้งหมด)
บางส่วน (Partial)ขึ้นกับแค่บางส่วนของคีย์ผสม → ปัญหาที่ 2NF จัดการ
ส่งผ่าน (Transitive)non-key ขึ้นกับ non-key อีกตัว → ปัญหาที่ 3NF จัดการ
เชิงกลุ่ม (MVD)การขึ้นต่อกันแบบหลายค่า → ปัญหาที่ 4NF จัดการ

ตอนที่ 5.2 วิธีการนอร์มัลไลเซชัน (6 ระดับ)

ทำเป็นลำดับ แต่ละระดับต้องผ่านระดับก่อนหน้าก่อน:

ระดับเงื่อนไข (เพิ่มจากระดับก่อนหน้า)
1NFทุกช่องเก็บค่าเดี่ยว (atomic) ไม่มีกลุ่มข้อมูลซ้ำ (no repeating group)
2NF1NF + ไม่มี partial dependency (non-key ขึ้นกับคีย์หลักทั้งหมด)
3NF2NF + ไม่มี transitive dependency
BCNFทุกดีเทอร์มิแนนต์ต้องเป็นคีย์ (candidate key)
4NFไม่มีการขึ้นต่อกันเชิงกลุ่ม (no MVD)
5NFไม่มีการขึ้นต่อกันแบบจอยน์ (no join dependency)

จำ: 2NF=ตัด Partial | 3NF=ตัด Transitive | BCNF=determinant ต้องเป็นคีย์ | 4NF=ตัด MVD | 5NF=ตัด Join — การทำถึง 3NF ขจัดความซ้ำซ้อนได้เกือบหมด

แนวตอบกิจกรรมระหว่างเรียน
5.1.1 นอร์มัลไลเซชันคืออะไร / วัตถุประสงค์
ทฤษฎีออกแบบฐานข้อมูล จัดการรีเลชันที่ซ้ำซ้อนให้อยู่ในรูปแบบบรรทัดฐาน เพื่อลดความซ้ำซ้อนและรักษาความถูกต้อง
5.1.2 ขั้นตอนการนอร์มัลไลเซชันมีกี่ขั้น
6 ขั้น: 1NF, 2NF, 3NF, BCNF, 4NF, 5NF
5.1.3 ดีเทอร์มิแนนต์และดีเพนเดนซีคืออะไร
ดีเทอร์มิแนนต์ = แอตทริบิวต์ที่ระบุค่าของแอตทริบิวต์อื่นได้; ดีเพนเดนซี = แอตทริบิวต์ที่ถูกระบุค่า (A → B)
5.2.1 คุณสมบัติของ 1NF
แต่ละช่องเก็บค่าเดี่ยว (atomic) ไม่กระจายได้อีก และไม่มีกลุ่มข้อมูลซ้ำ
5.2.2 คุณสมบัติของ 2NF
เป็น 1NF + non-key ต้องขึ้นกับคีย์หลักอย่างแท้จริง (no partial dependency)
5.2.3 คุณสมบัติของ 3NF
เป็น 2NF + non-key ต้องไม่ขึ้นกับ non-key ด้วยกัน (no transitive dependency)
5.2.4 คุณสมบัติของ BCNF
แอตทริบิวต์ที่เป็นดีเทอร์มิแนนต์ต้องเป็นคีย์ของรีเลชัน
5.2.5 คุณสมบัติของ 4NF
ต้องไม่มีการขึ้นต่อกันเชิงกลุ่ม (no MVD)
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดเป็นหลักการนอร์มัลไลเซชัน
เฉลย: จ. จัดการรีเลชันที่ซ้ำซ้อน ให้อยู่ในรูปแบบที่ง่ายต่อการนำไปใช้ — คือการปรับรีเลชันที่ซ้ำซ้อนให้อยู่ในรูปแบบบรรทัดฐาน
ข้อ 2. ข้อใดไม่ใช่ปัญหาที่เกิดจากความซ้ำซ้อนของข้อมูล
เฉลย: ก. การสืบค้นข้อมูล — ปัญหาจากความซ้ำซ้อน = เพิ่ม/แก้ไข/ลบ; การสืบค้นไม่ใช่ปัญหา
ข้อ 3. Normalization มีทั้งหมดกี่ระดับ
เฉลย: ง. 6 — 1NF, 2NF, 3NF, BCNF, 4NF, 5NF
ข้อ 4. (แผนภาพ) FD4 เป็นการขึ้นต่อกันแบบใด
เฉลย: ข. การขึ้นต่อกันแบบบางส่วน (Partial) — ขึ้นกับแค่บางส่วนของคีย์ผสม = partial dependency
ข้อ 5. (CustNo ซ้ำ) รีเลชันนี้ยังไม่ผ่านบรรทัดฐานระดับใด
เฉลย: ก. 1NF — มีกลุ่มข้อมูลซ้ำ/ไม่เป็นค่าเดี่ยว จึงยังไม่ผ่าน 1NF
ข้อ 6. (จากข้อ 5) มีการขึ้นต่อกันแบบใด
เฉลย: ข. การขึ้นต่อกันแบบบางส่วน (Partial) — CustName/City ขึ้นกับ CustNo ซึ่งเป็นบางส่วนของคีย์
ข้อ 7. (stdNo, sjNo, grade) อยู่ในบรรทัดฐานระดับใด
เฉลย: จ. 5NF — grade ขึ้นกับคีย์ผสมทั้งหมด ไม่มี partial/transitive/MVD = 5NF
ข้อ 8. นอร์มัลไลซ์ถึงระดับใดจึงขจัดความซ้ำซ้อนได้เกือบหมด
เฉลย: ค. 3NF — ทำถึง 3NF กำจัดความซ้ำซ้อนส่วนใหญ่ได้แล้ว
ข้อ 9. 1NF มีข้อเสียเรื่องการแก้/ลบ/เพิ่มข้อมูล ยกเว้นข้อใด
เฉลย: ค. — ตัวเลือก ค ไม่ใช่ข้อเสียของ 1NF (ดูตัวเลือกในเล่ม)
ข้อ 10. รีเลชัน 4NF ต้องมีคุณสมบัติใด
เฉลย: ค. ไม่มีการขึ้นต่อกันเชิงกลุ่ม (no MVD) — 4NF = ไม่มี multivalued dependency
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดคือเป้าหมายหลักของการนอร์มัลไลเซชัน
เฉลย: ก. ลดความซ้ำซ้อนของข้อมูล และรักษาความถูกต้อง — เป้าหมายหลัก = ลดความซ้ำซ้อน + รักษาความถูกต้อง
ข้อ 2. นอร์มัลไลซ์ถึงระดับใดจึงขจัดความซ้ำซ้อนได้เกือบหมด
เฉลย: ค. 3NF — ทำถึง 3NF กำจัดความซ้ำซ้อนได้เกือบหมด
ข้อ 3. ข้อใดคือการขึ้นต่อกันแบบเต็มค่า (Full FD)
เฉลย: ข. ฝั่งขวาขึ้นกับค่าของทุกแอตทริบิวต์ฝั่งซ้าย — Full = ขึ้นกับคีย์ผสมทั้งหมด ไม่ใช่บางส่วน
ข้อ 4. (รหัสวิชา-หนังสือ-รหัสผู้แต่ง) เป็นความสัมพันธ์แบบใด
เฉลย: ก. การขึ้นต่อกันแบบเชิงกลุ่ม (MVD) — วิชาหนึ่งมีหลายหนังสือและหลายผู้แต่งอิสระต่อกัน = MVD
ข้อ 5. รีเลชัน 1NF ต้องมีคุณสมบัติใด
เฉลย: ง. ไม่เกิดกลุ่มของข้อมูลที่วนซ้ำ (no repeating group) — 1NF = ค่าเดี่ยว ไม่มีกลุ่มข้อมูลซ้ำ
ข้อ 6. (studentID...subjectNo ซ้ำ) ไม่อยู่ในบรรทัดฐานระดับใด
เฉลย: ก. 1NF — มีกลุ่มข้อมูลซ้ำ (subjectNo หลายค่าต่อนักศึกษา) จึงยังไม่ผ่าน 1NF
ข้อ 7. ข้อใดไม่ใช่คุณสมบัติของรีเลชันใน 2NF
เฉลย: ค. ไม่มี transitive dependency — no transitive เป็นคุณสมบัติของ 3NF (2NF = no partial)
ข้อ 8. แอตทริบิวต์ต้องไม่เกิด transitive dependency เป็นคุณสมบัติของระดับใด
เฉลย: ข. 3NF — no transitive dependency = คุณสมบัติของ 3NF
ข้อ 9. (รหัสลูกค้า-taxID-ที่อยู่) ไม่อยู่ในบรรทัดฐานระดับใด
เฉลย: จ. BCNF — มี non-key (taxID) เป็นดีเทอร์มิแนนต์ระบุค่าคีย์ได้ จึงยังไม่ผ่าน BCNF
ข้อ 10. (จากข้อ 9) ปรับปรุงรีเลชันให้อยู่ในบรรทัดฐานได้อย่างไร
เฉลย: ง. ข้อ ก. และ ข. ถูก — แตกเป็น 2 รีเลชันเพื่อแยกการขึ้นต่อกันที่ทำให้ผิด BCNF (ทำได้หลายวิธีที่สมมูล)
สรุปประเด็นที่ออกสอบบ่อย
  • เป้าหมาย: ลดความซ้ำซ้อน + รักษาความถูกต้อง
  • Anomalies: เพิ่ม/แก้ไข/ลบ (ไม่ใช่การสืบค้น)
  • จำนวนระดับ: 6 (1NF/2NF/3NF/BCNF/4NF/5NF)
  • 1NF: ค่าเดี่ยว ไม่มี repeating group
  • 2NF: ตัด Partial dependency
  • 3NF: ตัด Transitive dependency (ขจัดซ้ำซ้อนได้เกือบหมด)
  • BCNF: ทุก determinant ต้องเป็นคีย์  ·  4NF/5NF: ตัด MVD / Join dependency

6.2.1 DDL — นิยาม/จัดการโครงสร้าง

CREATE TABLE employee ( emp_id CHAR(4) PRIMARY KEY, name CHAR(30), salary INT );
ALTER TABLE employee ADD email VARCHAR(50);

CHAR(n) = ความยาวคงที่ n (เติมช่องว่างเต็ม) · VARCHAR(n) = ความยาวไม่คงที่ (เก็บเท่าที่ใช้จริง)

6.2.2 DML — จัดการข้อมูล + JOIN

INSERT INTO employee VALUES ('E001','สมชาย',25000);

DELETE FROM employee WHERE emp_id = 'E001';

JOIN เชื่อมหลายตาราง: INNER JOIN (เฉพาะที่ตรงกัน), OUTER JOIN (LEFT/RIGHT/FULL — รวมที่ไม่ตรงด้วย)

SELECT e.name, d.dept_name
FROM employee e INNER JOIN department d ON e.dept_id = d.dept_id;

6.2.3 DCL — ควบคุมสิทธิ์

คำสั่งหน้าที่
GRANTให้สิทธิ์ผู้ใช้ (SELECT, UPDATE, ALL PRIVILEGES)

แนวตอบกิจกรรมระหว่างเรียน
6.1.1 ภาษา SQL จัดการอะไรได้ / แบ่งเป็นกี่กลุ่ม
จัดการได้ทั้งนิยามข้อมูลและสืบค้นข้อมูล แบ่งเป็น: นิยามข้อมูล (DDL), จัดการข้อมูล (DML), ควบคุมข้อมูล (DCL)
6.1.2 คำสั่ง SELECT / AS / DISTINCT
SELECT ดึงข้อมูลจากคอลัมน์; AS ตั้งชื่อคอลัมน์ผลลัพธ์; DISTINCT แสดงค่าไม่ซ้ำ
6.1.3 WHERE operators / ฟังก์ชันนับ / aggregate
WHERE มี 3 กลุ่ม operator; ฟังก์ชันนับ = COUNT; aggregate ใช้กับ GROUP BY
6.2.1 คำสั่ง DDL / CHAR vs VARCHAR
DDL = CREATE, ALTER, DROP; CHAR ขนาดคงที่, VARCHAR ขนาดไม่คงที่ เก็บเท่าที่ใช้
6.2.2 คำสั่ง DML / JOIN
DML = SELECT, INSERT, DELETE, UPDATE; JOIN เชื่อมหลายตาราง: INNER JOIN และ OUTER JOIN (LEFT/RIGHT/FULL)
6.2.3 คำสั่ง DCL / ส่งต่อสิทธิ์ / ยกเลิก
DCL = GRANT, REVOKE; ส่งต่อสิทธิ์ใช้ WITH GRANT OPTION; ยกเลิกสิทธิ์แบบลูกโซ่ใช้ CASCADE
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดคือกลุ่มของภาษา SQL
เฉลย: จ. ถูกทุกข้อ — SQL มีกลุ่มนิยามข้อมูล/จัดการ/ควบคุมข้อมูล/ควบคุมรายการ
ข้อ 2. กลุ่มภาษา SQL ที่ใช้ค้นหา/เพิ่ม/อัพเดท/ลบข้อมูล
เฉลย: ก. ภาษาสำหรับจัดการ (DML) — SELECT/INSERT/UPDATE/DELETE = DML
ข้อ 3. สัญลักษณ์ใดใช้เรียกดูข้อมูลทั้งตาราง
เฉลย: ก. เครื่องหมายดอกจัน (*) — SELECT * = แสดงทุกคอลัมน์ทั้งตาราง
ข้อ 4. ฟังก์ชันสำเร็จที่ใช้กับกลุ่มข้อมูลตัวเลข ให้ผล 1 คอลัมน์
เฉลย: จ. ถูกทุกข้อ — COUNT/SUM/AVG/MIN ล้วนเป็น aggregate function
ข้อ 5. ข้อใดเป็นคำสั่งของภาษาสำหรับจัดการ (DML)
เฉลย: จ. ถูกทุกข้อ — UPDATE/SELECT/INSERT/DELETE ล้วนเป็น DML
ข้อ 6. ข้อใดไม่สามารถใช้ในคำสั่ง WHERE เป็นเงื่อนไขค้นหา
เฉลย: ง. เปอร์เซ็นต์ (%) — % เป็น wildcard ของ LIKE ไม่ใช่ตัวเปรียบเทียบเดี่ยวๆ
ข้อ 7. คำสั่งจัดเรียงลำดับผลลัพธ์จากมากไปน้อย
เฉลย: ข. ORDER BY NAME DESC — ORDER BY ... DESC = เรียงมากไปน้อย
ข้อ 8. คำสั่งที่ใช้จัดเรียงข้อมูลเป็นกลุ่มตามเงื่อนไข
เฉลย: ข. GROUP BY — GROUP BY จัดกลุ่มข้อมูลตามคอลัมน์
ข้อ 9. ข้อใดไม่ใช่คำสั่งในกลุ่มภาษาควบคุมข้อมูล (DCL)
เฉลย: จ. INSERT — INSERT เป็น DML; GRANT/ALL PRIVILEGES/PUBLIC/WITH GRANT OPTION อยู่ใน DCL
ข้อ 10. ข้อใดทำให้ผู้ใช้ใช้คำสั่ง SQL จัดการ/สืบค้นไม่ได้
เฉลย: ก. ได้รับ REVOKE ALL PRIVILEGES — REVOKE ALL PRIVILEGES = เพิกถอนสิทธิ์ทั้งหมด
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. กลุ่มภาษา SQL ที่ใช้เปลี่ยนแปลง/ลบโครงสร้างข้อมูล
เฉลย: ก. กลุ่มภาษานิยามข้อมูล (DDL) — เปลี่ยน/ลบโครงสร้าง = DDL (CREATE/ALTER/DROP)
ข้อ 2. ข้อใดเป็นรูปแบบภาษาควบคุมข้อมูล (DCL)
เฉลย: ค. ใช้ควบคุมสิทธิ์การใช้งานของผู้ใช้ — DCL ควบคุมสิทธิ์การเข้าถึง (GRANT/REVOKE)
ข้อ 3. สัญลักษณ์ใดไม่สามารถใช้ในภาษา SQL
เฉลย: จ. ทุกข้อใช้ในภาษา SQL ได้ — ดอกจัน/เซมิโคลอน/คอมม่า/เท่ากับ ใช้ได้ทั้งหมด
ข้อ 4. ฟังก์ชันที่ใช้นับจำนวนพนักงานในแต่ละฝ่าย
เฉลย: ก. COUNT — นับจำนวน = COUNT (มักใช้กับ GROUP BY)
ข้อ 5. คำสั่ง DML ที่ใช้คิวรี/เรียกดูข้อมูล
เฉลย: ข. SELECT — เรียกดู/สืบค้น = SELECT
ข้อ 6. operator ใน WHERE สำหรับผลลัพธ์ย่อยหลายแถว
เฉลย: ค. อิน (IN) — IN ใช้ตรวจค่าที่อยู่ในชุด/ผล subquery หลายแถว
ข้อ 7. คำสั่ง SELECT * FROM user หมายความว่าอย่างไร
เฉลย: ค. แสดงผลข้อมูลทั้งหมดจากตาราง user — * = ทุกคอลัมน์
ข้อ 8. เครื่องหมายใดใช้คั่นระหว่างชื่อฟิลด์ในคำสั่ง SELECT
เฉลย: ข. เครื่องหมายคอมม่า (,) — คั่นชื่อคอลัมน์ด้วยคอมม่า
ข้อ 9. คำสั่ง DCL ที่ให้ผู้ใช้ส่งต่อสิทธิ์ที่ตนได้รับให้ผู้อื่น
เฉลย: ง. WITH GRANT OPTION — อนุญาตให้ผู้รับสิทธิ์ส่งต่อสิทธิ์ให้คนอื่นได้
ข้อ 10. ผู้ใดสามารถใช้คำสั่ง GRANT กำหนดสิทธิ์ให้ผู้อื่น
เฉลย: ค. ผู้บริหารฐานข้อมูล (DBA) — DBA มีสิทธิ์ทุกอย่าง จึง GRANT ให้ผู้อื่นได้
สรุปประเด็นที่ออกสอบบ่อย
  • กลุ่ม SQL: DDL / DML / DCL / TCL
  • DDL: CREATE / ALTER / DROP  ·  DML: SELECT / INSERT / UPDATE / DELETE  ·  DCL: GRANT / REVOKE
  • SELECT: * = ทุกคอลัมน์, AS = ตั้งชื่อ, DISTINCT = ไม่ซ้ำ
  • WHERE: =, <>, >, <, IN, BETWEEN, LIKE (% คู่กับ LIKE)
  • จัดกลุ่ม/เรียง: GROUP BY + HAVING, ORDER BY ASC/DESC
  • DCL ส่งต่อ/ถอน: WITH GRANT OPTION / REVOKE ... CASCADE
  • CHAR vs VARCHAR: คงที่ vs ไม่คงที่

การออกแบบฐานข้อมูลระดับกายภาพ

ภาพรวมของหน่วย — การออกแบบฐานข้อมูล "ระดับกายภาพ" (Physical) ขั้นสุดท้ายของการออกแบบ 3 ระดับ แบ่งเป็น 2 ตอน:
• ตอน 7.1 แนวคิด (ความหมาย, เลือก DBMS, แปลงตรรกะ→กายภาพ, กฎความคงสภาพ)
• ตอน 7.2 การนำไปใช้ (คำนวณเนื้อที่, โครงสร้างแฟ้ม 3 แบบ, รักษาความปลอดภัย)

ตอนที่ 7.1 แนวคิดการออกแบบฐานข้อมูลระดับกายภาพ

การออกแบบ 3 ระดับ: ① แนวคิด (วาด ER ยังไม่สนซอฟต์แวร์) → ② ตรรกะ (แปลง ER เป็นตาราง + นอร์มัลไลซ์) → ③ กายภาพ (ปรับให้ตรงกับ DBMS จริง: ชนิดข้อมูล วิธีเข้าถึง ความปลอดภัย เนื้อที่จัดเก็บ) — จึงต้องเลือกซอฟต์แวร์ก่อนออกแบบระดับนี้

แนวทางเลือกซอฟต์แวร์ DBMS — 5 ข้อ

  1. คุณลักษณะและเครื่องมือ (ออกแบบหน้าจอ/รายงาน/พจนานุกรมข้อมูล)
  2. ค่าใช้จ่าย (ลิขสิทธิ์/ซ่อมบำรุง/ฝึกอบรม)
  3. ความสามารถทำงานข้ามแพลตฟอร์ม
  4. ชนิดโครงสร้างของฐานข้อมูล (ลำดับชั้น/สัมพันธ์/วัตถุ)
  5. ความต้องการทางฮาร์ดแวร์ของผลิตภัณฑ์

กฎความคงสภาพของข้อมูล (integrity rule) — 2 กฎ

เอนทิตี้ (entity integrity): คีย์หลักต้องไม่ซ้ำ (unique) และไม่เป็นค่าว่าง (not null) · การอ้างอิง (referential integrity): ค่าคีย์นอก (FK) ต้องปรากฏเป็นคีย์หลักของอีกรีเลชัน หรือเป็นค่าว่าง (NULL)

คำสั่ง SQL กำหนดเงื่อนไข (CONSTRAINT)

CREATE TABLE Student (
  student_id CHAR(8) NOT NULL,
  student_name VARCHAR(30),
  sex CHAR(1),
  major_id CHAR(2),
  CONSTRAINT stID_pk PRIMARY KEY (student_id),
  CONSTRAINT sex_chk CHECK (sex IN ('M','F')),
  CONSTRAINT mjID_fk FOREIGN KEY (major_id)
    REFERENCES Major(major_id) ON DELETE SET NULL);

ตอนที่ 7.2 การนำไปใช้งาน

7.2.1 การคำนวณเนื้อที่จัดเก็บข้อมูล — สูตรออกสอบทุกครั้ง

  • เนื้อที่คอลัมน์ = ขนาดคอลัมน์ + เนื้อที่ที่ระบบใช้ต่อคอลัมน์
  • เนื้อที่แถว = ผลรวมทุกคอลัมน์ + เนื้อที่ที่ระบบใช้ต่อแถว
  • เนื้อที่ตาราง = (เนื้อที่แถว × จำนวนแถว) + เนื้อที่ดัชนี + เนื้อที่ที่ระบบใช้ต่อตาราง
  • เนื้อที่ฐานข้อมูล = ผลรวมเนื้อที่ของทุกตาราง

ค่าที่ต้องจำ: char 1 ตัว = 1 ไบต์, ตัวเลข 1 ตัว = 4 ไบต์, 1 KB = 1,024 ไบต์ · เนื้อที่สำรอง = อัตราการเพิ่ม(%) × เนื้อที่ฐานข้อมูล · ตัวอย่าง: Employee ต่อแถว 58 ไบต์ × 2,000 + ระบบ 16 KB ≈ 132 KB; + Department 17 KB = ฐานข้อมูล 149 KB

7.2.2 โครงสร้างแฟ้มข้อมูล 3 แบบ

แบบคุณสมบัติ / ข้อเสีย
เรียงลำดับ (sequential)อ่านต่อเนื่องตั้งแต่เรคอร์ดแรก เหมาะงานอ่านปริมาณมาก (ใบแจ้งหนี้) สื่อ=เทปแม่เหล็ก ราคาถูก · ข้อเสีย: ต้องเรียงก่อน ค้นช้า ไม่เหมาะแก้/เพิ่ม/ลบบ่อย
สุ่ม (direct/random — hashing)เข้าถึงตรงทันที เร็วสุด เหมาะธุรกรรมออนไลน์ สูตร Address = key MOD ขนาดตาราง · ข้อเสีย: ไม่เหมาะอ่านปริมาณมาก เขียนโปรแกรมซับซ้อน
ลำดับเชิงดัชนี (index sequential)รวมข้อดี 2 แบบ ค้นผ่านแฟ้มดัชนี ยืดหยุ่น เหมาะข้อมูลจำนวนมาก · ข้อเสีย: เปลืองเนื้อที่ดัชนี ช้ากว่าแบบสุ่ม

7.2.3 การรักษาความมั่นคงปลอดภัย — 3 วิธี

  • ตารางเสมือน (view): ให้ผู้ใช้เห็นเฉพาะบางส่วน (รายละเอียดในหน่วย 11)
  • กำหนดสิทธิ์ผู้ใช้: ยืนยันตัวบุคคล (รหัสผ่าน/สมาร์ตการ์ด/ไบโอเมตริก) + GRANT/REVOKE
  • เทคโนโลยี RAID: RAID 0 = striping (ไม่สำรอง) | RAID 1 = mirror สำเนา 2 ชุด | RAID 2 = ดิสก์เฉพาะเก็บ ECC ตรวจ/แก้ข้อผิดพลาด

7.1.2 ออกแบบโครงสร้างกายภาพให้ตาราง / กฎความคงสภาพ / SQL

7.2.1 คำนวณเนื้อที่ฐานข้อมูล / ผลกระทบเมื่อข้อมูลเพิ่มมาก

7.2.2 หลักการเข้าถึงแฟ้ม 3 แบบ / Hash table

แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)

เฉลย: ง — "ดัชนีค้นได้เร็วกว่าแบบสุ่ม" ผิด — ลำดับเชิงดัชนีช้ากว่าแบบสุ่ม
ข้อ 9. ข้อใดกล่าวไม่ถูกต้องเกี่ยวกับแฟ้มข้อมูลแบบลำดับเชิงดัชนี
เฉลย: ค — ตามเฉลย (ดูตัวเลือกในเล่ม)
ข้อ 10. ฮาร์ดดิสก์เฉพาะเก็บ ECC ตรวจ/แก้ข้อผิดพลาด จัดเป็น RAID ระดับใด
เฉลย: ค (RAID 2) — RAID 2 ใช้ ECC บนดิสก์เฉพาะ

ข้อ 2. ข้อใดไม่ใช่แนวทางพิจารณาเลือกซอฟต์แวร์ DBMS

ข้อ 3. ข้อใดไม่ใช่การกำหนดรายละเอียดให้กับข้อมูลในตาราง

ข้อ 4. กำหนดให้ฟิลด์ Employee_ID เป็น primary key
เฉลย: จ — CONSTRAINT Em_pk PRIMARY KEY (Employee_ID)
ข้อ 5. กำหนดขอบเขต STATUS ให้มี 2 ค่า P หรือ F

ข้อ 6. เนื้อที่ของตาราง Student มีค่าเท่าใด

ข้อ 7. เนื้อที่ของฐานข้อมูลมีค่าเท่าใด

ข้อ 8. ข้อใดกล่าวไม่ถูกต้องเกี่ยวกับโครงสร้างแฟ้มข้อมูล

ข้อ 9. ข้อใดกล่าวไม่ถูกต้องเกี่ยวกับแฟ้มข้อมูลแบบสุ่ม
เฉลย: ง — "แบบสุ่มเหมาะกับงานอ่านต่อเนื่องปริมาณมาก" ผิด — นั่นคือแบบเรียงลำดับ
ข้อ 10. disk mirror สำเนาข้อมูล 2 ชุดเหมือนกัน จัดเป็น RAID ระดับใด
เฉลย: ข (RAID 1) — RAID 1 = mirroring สำเนาข้อมูล 2 ชุด
สรุปประเด็นที่ออกสอบบ่อย
  • การออกแบบ 3 ระดับ: แนวคิด→ตรรกะ→กายภาพ; กายภาพต่างตรง "ผูกกับซอฟต์แวร์ DBMS จริง"
  • กฎความคงสภาพ 2 กฎ: เอนทิตี้ (PK ไม่ซ้ำ/ไม่ว่าง) + การอ้างอิง (FK ต้องมีจริง หรือ NULL)
  • CONSTRAINT: PRIMARY KEY / FOREIGN KEY…REFERENCES…ON DELETE SET NULL / CHECK…IN
  • สูตรเนื้อที่ 4 ชั้น: คอลัมน์→แถว→ตาราง→ฐานข้อมูล; 1 KB = 1,024 ไบต์
  • แฟ้ม 3 แบบ: เรียงลำดับ(เทป/อ่านต่อเนื่อง), สุ่ม-hashing(เร็ว/ออนไลน์ key MOD size), ลำดับเชิงดัชนี(รวม 2 แบบ)
  • RAID: 0=striping(ไม่สำรอง), 1=mirror(สำเนา), 2=ECC(ดิสก์เฉพาะตรวจ/แก้ error)

กรณีศึกษาการออกแบบฐานข้อมูล

ภาพรวมของหน่วย — หน่วย "ลงมือทำจริง" นำกระบวนการออกแบบจากหน่วย 3–7 มาประยุกต์กับ 2 กรณีศึกษา ตามลำดับ 4 ขั้น: วิเคราะห์เอนทิตี้/ER → แปลงเป็นสคีมา → นอร์มัลไลซ์ → เขียน SQL
• ตอน 8.1 ระบบยืมหนังสือห้องสมุด  ·  ตอน 8.2 ระบบธุรกิจจำหน่ายเฟอร์นิเจอร์

ตอนที่ 8.1 กรณีศึกษาระบบยืมหนังสือห้องสมุด

ขั้นตอนออกแบบ (4 ขั้น): วิเคราะห์เอนทิตี้/แอตทริบิวต์ → เขียน ER → แปลง ER เป็นสคีมา → นอร์มัลไลซ์ → เขียน SQL

เอนทิตี้ vs แอตทริบิวต์ (ออกสอบชัวร์)

  • เอนทิตี้: BOOK, MAGAZINE, STUDENT
  • ISBN = แอตทริบิวต์ของ BOOK (ไม่ใช่เอนทิตี้)
  • publisher (สำนักพิมพ์) = แอตทริบิวต์ที่อยู่ทั้งใน BOOK และ MAGAZINE
  • ครูประจำชั้น (teacher) = แอตทริบิวต์ของ STUDENT

ความสัมพันธ์ + นอร์มัลไลเซชัน: STUDENT ยืม BOOK/MAGAZINE (นักเรียน 1 คน ยืมได้หลายเล่ม) · ถ้ามี FD mname → publisher, mcategory (non-key ระบุ non-key) = transitive → ละเมิด 3NF ต้องแยกตาราง

DELETE FROM BOOK WHERE publisher = 'Prentice Hall';
UPDATE MAGAZINE SET mcategory = 'วิชาการ'
WHERE mname = 'ไอทีก้าวหน้า';

ตอนที่ 8.2 กรณีศึกษาระบบธุรกิจจำหน่ายเฟอร์นิเจอร์

เอนทิตี้ vs แอตทริบิวต์

  • เอนทิตี้: PRODUCT, PART, ORDERED, INVOICE
  • TYPE = แอตทริบิวต์ (ไม่ใช่เอนทิตี้)
  • min_q = แอตทริบิวต์ของ PART
  • prod_no, prod_des, prod_type, prod_color = แอตทริบิวต์ของ PRODUCT

ความสัมพันธ์ M:N → ต้องสร้างตารางเชื่อม + แอตทริบิวต์บนความสัมพันธ์: PRODUCT ↔ PART (ตาราง PROD_PART; q_use=จำนวนชิ้นส่วนที่ใช้) · PRODUCT ↔ ORDERED (ปริมาณ/จำนวนที่สั่งซื้อ) · INVOICE ↔ PART (ตาราง INV_PART; part_no เป็น FK อ้างอิงจาก PART)

นอร์มัลไลเซชัน (FD): prod_type → prod_color (non-key→non-key) = transitive → PRODUCT ไม่อยู่ใน 3NF ต้องแยกตาราง · FD vend_no, inv_no, part_no → q_add, paid = full dependency ปกติของตารางเชื่อม

DELETE FROM PRODUCT
WHERE prod_des = 'โต๊ะ' AND prod_type = 'ไม้สัก';

UPDATE ORDERED SET s_addr = '989 สายใน',
  s_dist = 'เมือง', s_prov = 'ปทุมธานี'
WHERE cname = 'สุดใจ มากมี';

8.1 วิเคราะห์เอนทิตี้/ER/สคีมา/นอร์มัลไลซ์/SQL ของระบบยืมหนังสือ
ทำตามขั้นตอน 4 ขั้น (ER → สคีมา → นอร์มัลไลซ์ → SQL); ในเล่มมีแผนภาพ ER และสคีมาเต็ม
8.2 วิเคราะห์ของระบบเฟอร์นิเจอร์
เหมือนกัน เน้นจุดที่มีความสัมพันธ์ M:N (PROD_PART, INV_PART) ต้องสร้างตารางเชื่อม และตรวจ FD ให้ถึง 3NF
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดไม่ใช่เอนทิตี้ (ระบบยืมหนังสือ)
เฉลย: ง — ISBN เป็นแอตทริบิวต์ของ BOOK ไม่ใช่เอนทิตี้
ข้อ 2. แอตทริบิวต์ครูประจำชั้น (teacher) อยู่ในเอนทิตี้ใด
เฉลย: จ — ครูประจำชั้นเป็นแอตทริบิวต์ของ STUDENT
ข้อ 3. ความสัมพันธ์ระหว่าง BOOK และ STUDENT
เฉลย: ง — นักเรียน 1 คน ยืมหนังสือได้หลายเล่ม
ข้อ 4. เลชัน BOOK อยู่ในนอร์มัลฟอร์มที่ 2 ได้ถูกต้อง
เฉลย: จ — 2NF = non-key ทุกตัวขึ้นกับคีย์หลักทั้งหมด (ไม่มี partial dependency)
ข้อ 5. ลบข้อมูลสำนักพิมพ์ Prentice Hall ออกจาก BOOK
เฉลย: ข — DELETE FROM BOOK WHERE publisher = 'Prentice Hall'
ข้อ 6. ข้อใดไม่ใช่เอนทิตี้ (ระบบเฟอร์นิเจอร์)
เฉลย: จ — TYPE เป็นแอตทริบิวต์ (PART/INVOICE/ORDERED/PRODUCT เป็นเอนทิตี้)
ข้อ 7. แอตทริบิวต์ min_q อยู่ในเอนทิตี้ใด
เฉลย: ง — min_q เป็นแอตทริบิวต์ของ PART
ข้อ 8. แอตทริบิวต์บนความสัมพันธ์ PRODUCT และ ORDERED
เฉลย: จ — M:N ทำให้เกิดแอตทริบิวต์ "ปริมาณ/จำนวนที่สั่งซื้อ"
ข้อ 9. FD ใดทำให้เลชัน PRODUCT ไม่อยู่ใน 3NF
เฉลย: ข — prod_type → prod_color เป็น transitive dependency → ละเมิด 3NF
ข้อ 10. ปรับปรุงที่อยู่ลูกค้า "สุดใจ มากมี"
เฉลย: ค — UPDATE ORDERED SET ... WHERE cname=’สุดใจ มากมี’
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดเป็นแอตทริบิวต์ (ระบบยืมหนังสือ)
เฉลย: ข — ISBN เป็นแอตทริบิวต์ (BOOK/MAGAZINE/STUDENT เป็นเอนทิตี้)
ข้อ 2. แอตทริบิวต์ publisher อยู่ในเอนทิตี้ใด
เฉลย: ง — publisher อยู่ทั้งใน BOOK และ MAGAZINE
ข้อ 3. ความสัมพันธ์ระหว่าง MAGAZINE และ STUDENT
เฉลย: ข — นักเรียน 1 คน ยืมวารสารได้หลายเล่ม
ข้อ 4. FD ใดทำให้เลชัน MAGAZINE ไม่อยู่ใน 3NF
เฉลย: ค — mname → publisher, mcategory เป็น transitive → ละเมิด 3NF
ข้อ 5. ปรับปรุง mcategory ของวารสาร "ไอทีก้าวหน้า"

ข้อ 6. ข้อใดเป็นแอตทริบิวต์ (ระบบเฟอร์นิเจอร์)
เฉลย: จ — ทุกข้อเป็นเอนทิตี้ จึงไม่มีข้อใดเป็นแอตทริบิวต์
ข้อ 7. แอตทริบิวต์บนความสัมพันธ์ PRODUCT และ PART
เฉลย: ค — q_use (จำนวนชิ้นส่วนที่ใช้) เป็นแอตทริบิวต์บนความสัมพันธ์ M:N
ข้อ 8. part_no (FK) ใน INV_PART อ้างอิงจากตารางใด
เฉลย: ค — part_no อ้างอิงมาจากตาราง PART
ข้อ 9. FD vend_no, inv_no, part_no → q_add, paid อธิบายถูกต้อง
เฉลย: จ — เมื่อทราบคีย์หลักร่วมครบ จะระบุค่า q_add และ paid ได้
ข้อ 10. ลบผลิตภัณฑ์ "โต๊ะ ไม้สัก" ออกจาก PRODUCT

สรุปประเด็นที่ออกสอบบ่อย
  • เอนทิตี้ vs แอตทริบิวต์: ISBN/publisher/TYPE/min_q = แอตทริบิวต์; BOOK/MAGAZINE/STUDENT/PRODUCT/PART/INVOICE/ORDERED = เอนทิตี้
  • M:N ต้องสร้างตารางเชื่อม (PROD_PART, INV_PART) + มีแอตทริบิวต์บนความสัมพันธ์ (q_use, ปริมาณสั่งซื้อ)
  • FD แบบ transitive (prod_type→prod_color, mname→publisher) = ละเมิด 3NF ต้องแยกตาราง
  • SQL: DELETE FROM …WHERE…; และ UPDATE …SET…WHERE…; (SET มาก่อน WHERE)
  • ระวัง ข้อมูลที่แยกหลายคอลัมน์ (prod_des + prod_type) → WHERE ต้องระบุครบ
9

การใช้งานซอฟต์แวร์ประยุกต์ฐานข้อมูล

ภาพรวมของหน่วย — ซอฟต์แวร์จัดการฐานข้อมูลจริง 2 ตัวเปรียบเทียบกัน แบ่งเป็น 2 ตอน:
• ตอน 9.1 มายเอสคิวแอล (MySQL): DBMS โอเพ่นซอร์ส
• ตอน 9.2 ซีเควลเซิร์ฟเวอร์ (SQL Server): DBMS ของไมโครซอฟท์

ตอนที่ 9.1 มายเอสคิวแอล (MySQL)

  • DBMS แบบโอเพ่นซอร์ส ทำงานตามมาตรฐาน SQL
  • mysqld = โปรแกรมฝั่งเซิร์ฟเวอร์ของ MySQL ทำงานในรูปแบบเซอร์วิส (service)
  • storage engine (ตัวจัดการพื้นที่จัดเก็บ): InnoDB, MyISAM, MEMORY, FEDERATED — ⚠ BUFFER ไม่ใช่ storage engine
  • ชนิดข้อมูลตัวเลข: INT, SMALLINT, TINYINT, BIGINT

คำสั่งจัดการตารางที่ออกสอบ

CREATE TABLE IF NOT EXISTS employee (…);  -- สร้างเฉพาะเมื่อยังไม่มีตารางนี้
ALTER TABLE employee ADD COLUMN performance INT AFTER position;  -- เพิ่มฟิลด์หลังฟิลด์ที่ระบุ
explain employee;  -- แสดงโครงสร้างตาราง
DELETE FROM student;  TRUNCATE TABLE student;  -- ลบเฉพาะข้อมูล (เก็บโครงสร้าง)
DROP TABLE employee;  -- ลบทั้งโครงสร้าง

ฟังก์ชัน TRIM(BOTH 'x' FROM 'xxxbarxxx') → ตัดอักขระ x หัวและท้าย ได้ผล 'bar'

ตอนที่ 9.2 ซีเควลเซิร์ฟเวอร์ (SQL Server)

  • Enterprise Edition = รุ่นสูงสุด เป้าหมายศูนย์ข้อมูลขนาดใหญ่ + คลังข้อมูล + ธุรกิจอัจฉริยะ (BI)
  • SSMS (SQL Server Management Studio) = ศูนย์รวมการจัดการระบบต่างๆ; เชื่อมต่อเลือก Server type = Database Engine
  • system DB 4 ตัว: master, model, msdb, tempdb (sqldb ไม่ใช่); model = เทมเพลตสำหรับสร้างฐานข้อมูลใหม่
  • ไฟล์ฐานข้อมูล: data file (.mdf) และ log file (.ldf)

T-SQL (Transact-SQL)

เป็นภาษาที่ขยายเพิ่มจาก SQL ลิขสิทธิ์ของไมโครซอฟท์ · ประกาศตัวแปรใช้ @ นำหน้า

DECLARE @x varchar(10);
SELECT @x;  -- แสดงค่าที่เก็บไว้ในตัวแปร @x
แนวตอบกิจกรรมระหว่างเรียน
9.1 พื้นฐาน/การจัดการฐานข้อมูล-ตาราง/ส่วนขยายของ MySQL
ติดตั้ง+ใช้งาน MySQL, mysqld เป็น service, ใช้คำสั่ง CREATE/ALTER/INSERT/UPDATE/DELETE และฟังก์ชันส่วนขยาย เช่น TRIM, IF NOT EXISTS
9.2 ความรู้เบื้องต้น SQL Server / T-SQL
รู้จัก edition, SSMS, system DB, ไฟล์ data/log และเขียน T-SQL ประกาศตัวแปร @ + SELECT แสดงค่า
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดเป็นตัวจัดการพื้นที่จัดเก็บ (storage engine) ของ MySQL
เฉลย: ก — InnoDB เป็น storage engine ของ MySQL
ข้อ 2. ข้อใดกล่าวถูกต้องเกี่ยวกับ mysqld
เฉลย: ข — mysqld = โปรแกรมฝั่งเซิร์ฟเวอร์ ทำงานเป็นเซอร์วิส
ข้อ 3. วิธีร้องขอใช้บริการเข้ามาที่ Server ของ MySQL
เฉลย: จ — ทำได้ทุกแบบ (local/remote client, remote login, web script)
ข้อ 4. ข้อใดไม่เป็นชนิดข้อมูลของ MySQL
เฉลย: จ — INT/SMALLINT/TINYINT/BIGINT ล้วนเป็นชนิดข้อมูล (จึงตอบ ถูกทุกข้อ)
ข้อ 5. เพิ่มฟิลด์ performance INT หลังฟิลด์ position
เฉลย: ง — ALTER TABLE employee ADD COLUMN performance INT AFTER position;
ข้อ 6. ข้อใดกล่าวถูกต้องเกี่ยวกับ Enterprise Edition
เฉลย: ก — Enterprise = รุ่นสูงสุด ศูนย์ข้อมูลขนาดใหญ่ + คลังข้อมูล + BI
ข้อ 7. ข้อใดกล่าวถูกต้องเกี่ยวกับ SSMS
เฉลย: ค — SSMS = ศูนย์รวมการจัดการระบบต่างๆ ของ SQL Server
ข้อ 8. ข้อใดไม่ใช่ฐานข้อมูลระบบของ SQL Server
เฉลย: ง — sqldb ไม่ใช่ system DB (master, model, msdb, tempdb)
ข้อ 9. ข้อใดกล่าวถูกต้องเกี่ยวกับ T-SQL
เฉลย: ข — T-SQL = ส่วนขยายของ SQL ลิขสิทธิ์ไมโครซอฟท์
ข้อ 10. ประกาศตัวแปร T-SQL เป็นข้อความ 10 ตัวอักษร
เฉลย: ง — DECLARE @x varchar(10); (ตัวแปร T-SQL ขึ้นต้นด้วย @)
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดไม่ใช่ storage engine ของ MySQL
เฉลย: จ — BUFFER ไม่ใช่ storage engine (InnoDB/MyISAM/MEMORY/FEDERATED เป็น)
ข้อ 2. โปรแกรมฝั่งเซิร์ฟเวอร์ของ MySQL ทำงานเป็นเซอร์วิส
เฉลย: ข — mysqld
ข้อ 3. CREATE TABLE IF NOT EXISTS — IF NOT EXISTS หมายถึง
เฉลย: ค — สร้างตารางต่อเมื่อยังไม่มีตารางนี้ในฐานข้อมูล
ข้อ 4. คำสั่ง MySQL ใดแสดงโครงสร้างตาราง (Field/Type/Null/Key…)
เฉลย: ง — explain employee;
ข้อ 5. ลบข้อมูลทั้งหมดของตาราง student (ลบเฉพาะข้อมูล)
เฉลย: จ — DELETE FROM student; และ TRUNCATE TABLE student; ทำได้ทั้งคู่
ข้อ 6. ผลของ SELECT TRIM(BOTH ’x’ FROM ’xxxbarxxx’)
เฉลย: ก — ตัด x หัวและท้ายออก เหลือ 'bar'
ข้อ 7. ฐานข้อมูลใดใช้เป็นเทมเพลตสำหรับสร้างฐานข้อมูลใหม่
เฉลย: ก — model
ข้อ 8. การเชื่อมต่อ SSMS ระบุ Server type เป็นข้อมูลใด
เฉลย: ค — Database Engine
ข้อ 9. ไฟล์พื้นฐานของ Database files ใน SQL Server
เฉลย: ง — data file และ log file
ข้อ 10. ผลของคำสั่ง SELECT @x ของ T-SQL
เฉลย: ง — แสดงค่าที่เก็บไว้ในตัวแปร @x
สรุปประเด็นที่ออกสอบบ่อย
  • MySQL: storage engine = InnoDB/MyISAM/MEMORY/FEDERATED (ไม่ใช่ BUFFER); mysqld = server service
  • MySQL คำสั่ง: IF NOT EXISTS, ALTER…ADD COLUMN…AFTER, explain, TRUNCATE/DELETE FROM (ลบข้อมูล), DROP (ลบโครงสร้าง)
  • SQL Server: Enterprise = รุ่นสูงสุด; SSMS = ศูนย์จัดการ (Server type = Database Engine)
  • system DB 4 ตัว: master/model/msdb/tempdb (model = เทมเพลต); ไฟล์ = data file + log file
  • T-SQL = ส่วนขยายลิขสิทธิ์ไมโครซอฟท์; ตัวแปรใช้ @ เช่น DECLARE @x varchar(10);
10

กฎควบคุมความถูกต้องของข้อมูลและพจนานุกรมข้อมูล

ภาพรวมของหน่วย — กฎที่ทำให้ข้อมูลในฐานข้อมูลถูกต้องเสมอ และพจนานุกรมข้อมูล แบ่งเป็น 3 ตอน:
• ตอน 10.1 กฎควบคุมความถูกต้องของ "สถานะข้อมูล" (โดเมน + การอ้างอิง)
• ตอน 10.2 กฎควบคุมในขั้น "การปรับปรุงข้อมูล" (assertion + trigger)
• ตอน 10.3 พจนานุกรมข้อมูล

ตอนที่ 10.1 กฎควบคุมความถูกต้องของสถานะข้อมูล

กฎควบคุมสถานะข้อมูล = 2 กฎ: กฎความถูกต้องของโดเมน (domain integrity) + กฎความถูกต้องของการอ้างอิง (referential integrity)

1) กฎความถูกต้องของโดเมน (domain)

ควบคุม 3 ส่วน: แอตทริบิวต์ที่เป็นคีย์หลัก · ชนิดและขนาดของแอตทริบิวต์ · โครงสร้างของแอตทริบิวต์ — คีย์เวิร์ด: PRIMARY KEY, NOT NULL, UNIQUE, CHECK

sex CHAR(1) CHECK (VALUE IN ('F','M'))
salary NUMERIC(8,2) CHECK (VALUE > 0 AND VALUE <= 150000)

2) กฎความถูกต้องของการอ้างอิง (referential)

ควบคุมคีย์นอก (FK) — คีย์เวิร์ดคือ REFERENCES · กำหนดการกระทำเพิ่มได้ เช่น Automatic, Customize, Default, Null

cus_id CHAR(5) NOT NULL REFERENCES customer(cus_id)

การตรวจ INSERT ว่าถูกกฎ: ต้องผ่านทุกเงื่อนไข (คีย์หลักห้าม null, ค่าต้องอยู่ในช่วง CHECK, เพศต้องอยู่ใน VALUE IN, ชนิด/ขนาดต้องตรง) — ผิดบ่อย: ssn เป็น null, sex=’A’, salary=0

ตอนที่ 10.2 กฎควบคุมในขั้นตอนการปรับปรุงข้อมูล

  • การประกาศยืนยัน (assertion): ประโยคตรวจสอบความถูกต้องทั้ง "ก่อนและหลัง" การปรับปรุงข้อมูล
  • ทริกเกอร์ (trigger): กฎที่สั่งให้ระบบกระทำการอย่างใดอย่างหนึ่งตามที่ผู้ใช้ต้องการ เมื่อตรวจพบข้อผิดพลาด

ตอนที่ 10.3 พจนานุกรมข้อมูล (Data Dictionary)

  • คือที่เก็บ "ข้อมูลของข้อมูล" (metadata) อธิบายรายละเอียดโครงสร้างตาราง
  • องค์ประกอบ: ชื่อข้อมูล, รายละเอียด, ชนิด, ขนาด/ความยาว (เช่น VARCHAR(40)), ชื่อตาราง, คีย์หลัก/คีย์นอก
  • ใช้ "กลุ่มภาษานิยามข้อมูล (DDL)" ระบุรายละเอียดโครงสร้างที่ได้จากการออกแบบ
แนวตอบกิจกรรมระหว่างเรียน
10.1 กฎควบคุมความถูกต้องของโดเมน / การอ้างอิง
โดเมนคุม คีย์หลัก/ชนิด-ขนาด/โครงสร้าง (PRIMARY KEY, NOT NULL, CHECK); การอ้างอิงคุม FK ด้วย REFERENCES
10.2 กฎควบคุมในขั้นตอนการปรับปรุงข้อมูล
assertion ตรวจก่อน/หลังปรับปรุง; trigger สั่งระบบกระทำการเมื่อพบข้อผิดพลาด
10.3 พจนานุกรมข้อมูล มีองค์ประกอบและประโยชน์อะไร
เก็บ metadata (ชื่อ/ชนิด/ขนาด/คีย์) ใช้ DDL ระบุ ช่วยให้เข้าใจโครงสร้างฐานข้อมูล
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดจัดเป็นกฎควบคุมสถานะข้อมูล
เฉลย: ก — กฎความถูกต้องของโดเมน และ การอ้างอิง
ข้อ 2. ข้อมูลใดควบคุมด้วยกฎความถูกต้องของโดเมน
เฉลย: จ — ถูกทุกข้อ (คีย์หลัก, ชนิด, ขนาด, โครงสร้างของแอตทริบิวต์)
ข้อ 3. การกำหนดกฎควบคุมของแอตทริบิวต์ที่เป็นคีย์หลัก
เฉลย: ข — std_id CHAR(10) PRIMARY KEY
ข้อ 4. การกำหนดกฎความถูกต้องของการอ้างอิง
เฉลย: จ — cus_id … REFERENCES customer(cus_id)
ข้อ 5. คำสั่ง INSERT ในข้อใดถูกต้องตามกฎ
เฉลย: ค — ค่าทุกฟิลด์ผ่านเงื่อนไข (ssn 13 หลัก, sex='M', salary อยู่ในช่วง)
ข้อ 6. คำสั่งในข้อใดผิดกฎควบคุมของแอตทริบิวต์ salary
เฉลย: จ — salary = 0 ผิด CHECK (VALUE > 0)
ข้อ 7. เงื่อนไขที่กำหนดเพิ่มในกฎความถูกต้องของการอ้างอิง
เฉลย: จ — ถูกทุกข้อ (Automatic/Customize/Default/Null)
ข้อ 8. กฎควบคุมในกระบวนการทำรายการ (transaction)
เฉลย: ง — ใช้ประโยคตรวจสอบทั้งก่อนและหลังการปรับปรุงข้อมูล (assertion)
ข้อ 9. พจนานุกรมข้อมูลแสดงรายละเอียดของตารางได้แก่
เฉลย: จ — ถูกทุกข้อ (ชื่อตาราง/คีย์หลัก/ข้อมูลผู้ใช้/คีย์นอก)
ข้อ 10. ข้อใดไม่ใช่องค์ประกอบของพจนานุกรมข้อมูล
เฉลย: จ — ทุกข้อ (ชื่อ/รายละเอียด/ชนิด/ขนาด) เป็นองค์ประกอบทั้งหมด
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดหมายถึงกฎควบคุมสถานะข้อมูล
เฉลย: ง — กฎที่กำหนดค่าต่างๆ เพื่อตรวจสอบความถูกต้องไว้ชัดเจน
ข้อ 2. ข้อมูลใดถูกควบคุมด้วยกฎความถูกต้องของการอ้างอิง
เฉลย: ข — คีย์นอก (FK)
ข้อ 3. ข้อใดไม่ใช่การกำหนดกฎควบคุมของโครงสร้างแอตทริบิวต์
เฉลย: ก — std_id … PRIMARY KEY เป็นการกำหนดคีย์หลัก ไม่ใช่โครงสร้าง
ข้อ 4. คำสำคัญใดใช้กำหนดกฎความถูกต้องของการอ้างอิง
เฉลย: ง — REFERENCES
ข้อ 5. คำสั่ง INSERT ในข้อใดถูกต้องตามกฎ
เฉลย: ก — ค่าทุกฟิลด์ผ่านเงื่อนไข (c_id 5 หลัก ไม่ null, credit อยู่ในช่วง 0–50000)
ข้อ 6. คำสั่งในข้อใดผิดกฎควบคุมของคีย์หลัก
เฉลย: ข — c_id เป็น null ผิดกฎคีย์หลัก (NOT NULL)
ข้อ 7. กำหนดให้เพิ่มค่าโดยปริยายลงในตารางหลักใช้รูปแบบใด
เฉลย: ค — Default (เพิ่มค่าโดยปริยาย)
ข้อ 8. กฎควบคุมด้วยการสร้างทริกเกอร์ (trigger)
เฉลย: จ — กฎที่สั่งให้ระบบกระทำการตามผู้ใช้ต้องการเมื่อพบข้อผิดพลาด
ข้อ 9. พจนานุกรมข้อมูลใช้คำสั่งกลุ่มใดระบุรายละเอียดโครงสร้าง
เฉลย: ก — กลุ่มภาษานิยามข้อมูล (DDL)
ข้อ 10. ลักษณะข้อมูลที่บอกขนาด/ความยาว (length)
เฉลย: จ — VARCHAR(40)
สรุปประเด็นที่ออกสอบบ่อย
  • กฎควบคุมสถานะข้อมูล = โดเมน (คีย์หลัก/ชนิด-ขนาด/โครงสร้าง: PRIMARY KEY, NOT NULL, CHECK) + การอ้างอิง (FK: REFERENCES)
  • ตรวจ INSERT: ผ่านทุกเงื่อนไขจึงถูก — ผิดบ่อย: คีย์หลัก null, ค่าเกินช่วง CHECK, เพศไม่อยู่ใน VALUE IN
  • กฎในขั้นปรับปรุง: assertion (ตรวจก่อน/หลัง) vs trigger (สั่งระบบกระทำการเมื่อพบข้อผิดพลาด)
  • การอ้างอิงกำหนดการกระทำเพิ่มได้: Default (เพิ่มค่าปริยาย), Null ฯลฯ
  • พจนานุกรมข้อมูล = metadata (ชื่อ/ชนิด/ขนาด/คีย์) ระบุด้วย DDL; ความยาว เช่น VARCHAR(40)
11

วิวและอินเด็กซ์

ภาพรวมของหน่วย — เครื่องมือ 2 อย่างที่ช่วยจัดการ/เพิ่มประสิทธิภาพฐานข้อมูล แบ่งเป็น 2 ตอน:
• ตอน 11.1 วิว (View): ตารางเสมือนสร้างจาก SELECT
• ตอน 11.2 อินเด็กซ์ (Index): ตัวช่วยค้นข้อมูลให้เร็วขึ้น

ตอนที่ 11.1 วิว (View)

  • วิว = ตารางเสมือน สร้างจากตารางข้อมูลด้วยคำสั่ง SELECT — ไม่มีข้อมูลเป็นของตนเอง
  • ประกอบด้วยแถวและคอลัมน์เหมือนตารางข้อมูล และเมื่อตารางหลักเปลี่ยน ข้อมูลในวิวเปลี่ยนตาม
  • สร้างจากตารางหลักได้มากกว่า 1 ตาราง

ประโยชน์: จำกัดการเข้าถึงข้อมูล (security — เลือกเฉพาะบางคอลัมน์ให้ผู้ใช้เห็น) · ลดความซับซ้อนของการเรียกใช้ข้อมูล (ใช้วิวแทน SELECT ที่ JOIN หลายตาราง)

ประเภทของวิว

  • วิวระดับ (แนวนอน/horizontal): เลือกเฉพาะ “แถว” ด้วย WHERE
  • วิวทางดิ่ง (vertical): เลือกเฉพาะ “คอลัมน์” บางคอลัมน์
  • วิวกลุ่ม: มี GROUP BY / ฟังก์ชันสรุปผล เช่น COUNT(*)
  • วิวข้อมูล (materialized): เก็บข้อมูลจริงไว้ ช่วยเพิ่มประสิทธิภาพเมื่อวิวซับซ้อนมาก — ต่างจากวิวเสมือน (virtual) ที่ไม่เก็บข้อมูล
CREATE VIEW view_job AS
  SELECT name, job_title, phone FROM employee;
GRANT SELECT, UPDATE ON view_job TO manager;

(ชื่อหลัง CREATE VIEW คือ ชื่อวิว ไม่ใช่ชื่อตารางหลัก; GRANT = กำหนดผู้มีสิทธิ์ใช้งานวิว)

ตอนที่ 11.2 อินเด็กซ์ (Index)

อินเด็กซ์ ทำงานเหมือนดัชนีท้ายเล่มหนังสือ / หมายเลขเรียกหนังสือในห้องสมุด — ช่วยค้นข้อมูลได้เร็วโดยไม่ต้องไล่ทีละแถว

ถูกสร้างเมื่อ: เป็นคีย์หลักของตาราง (อัตโนมัติ) · ใช้คำสั่ง CREATE INDEX · กำหนดคอลัมน์เป็น UNIQUE — อินเด็กซ์ผสม (composite) = ทำจากคอลัมน์มากกว่า 1 คอลัมน์

CREATE INDEX supplier_idx ON supplier_product (product_no, supplier_id);
CREATE INDEX order_date_idx ON "order" (order_date DESC);  -- ใหม่→เก่า

DESC = เรียงจากวันที่ล่าสุดก่อน (ใหม่→เก่า); ASC/ไม่ใส่ = เก่า→ใหม่

เกณฑ์ควร/ไม่ควรสร้างอินเด็กซ์

  • ควรสร้าง: ตารางมีเรคอร์ดจำนวนมาก, คอลัมน์ที่ใช้เป็นเงื่อนไข/จัดเรียงบ่อย, ค่าข้อมูลหลากหลาย
  • ไม่ควรสร้าง: ตารางที่เพิ่ม/ลบ/แก้ไขข้อมูลบ่อยๆ, คอลัมน์ค่าซ้ำมาก, คอลัมน์รูปภาพ/ขนาดใหญ่
แนวตอบกิจกรรมระหว่างเรียน
11.1 วิวคืออะไร / สร้างและใช้งานวิว
ตารางเสมือนจาก SELECT ไม่เก็บข้อมูลจริง; ใช้จำกัดการเข้าถึง + ลดความซับซ้อนคิวรี่; CREATE VIEW … AS SELECT …
11.2 อินเด็กซ์คืออะไร / สร้างและเลือกใช้
ตัวช่วยค้นเร็วเหมือนดัชนีหนังสือ; CREATE INDEX … ON ตาราง(คอลัมน์); สร้างบนตารางใหญ่/คอลัมน์ค้นบ่อย เลี่ยงตารางที่แก้ไขบ่อย
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดกล่าวถึงวิวได้ถูกต้อง
เฉลย: ค — วิวประกอบด้วยแถวและคอลัมน์เช่นเดียวกับตารางข้อมูล
ข้อ 2. ใช้วิวแทน SELECT ที่ JOIN หลายตาราง ช่วยเรื่องใดโดยตรง
เฉลย: ข — ลดความซับซ้อนของการเรียกใช้ข้อมูล
ข้อ 3. ข้อใดช่วยประสิทธิภาพเมื่อเรียกผ่านวิวที่ซับซ้อนมาก
เฉลย: จ — วิวข้อมูล (materialized) — เก็บข้อมูลจริงไว้ จึงเรียกใช้เร็ว
ข้อ 4. CREATE VIEW … WHERE region=’SOUTH’ เป็นวิวแบบใด
เฉลย: ก — วิวระดับ (เลือกเฉพาะแถวด้วย WHERE)
ข้อ 5. CREATE VIEW view_job … + GRANT … TO manager
เฉลย: จ — คำสั่งมีการกำหนดผู้มีสิทธิ์ใช้งานวิวด้วย (GRANT)
ข้อ 6. ข้อใดมีลักษณะการทำงานเหมือนอินเด็กซ์
เฉลย: ก — ดัชนีในหนังสือที่เรียงคำสำคัญพร้อมเลขหน้ากำกับ
ข้อ 7. อินเด็กซ์ถูกสร้างได้ในกรณีใด
เฉลย: ง — กรณีที่เป็นคีย์หลักของตาราง (สร้างอัตโนมัติ)
ข้อ 8. นำคอลัมน์มากกว่า 1 คอลัมน์มาทำอินเด็กซ์ เรียกว่า
เฉลย: ค — อินเด็กซ์ผสม (composite index)
ข้อ 9. คำสั่งสร้างอินเด็กซ์ใดช่วยรายงานเรียงวันที่ ล่าสุดก่อน
เฉลย: ก — CREATE INDEX … ON order(order_date DESC)
ข้อ 10. เกณฑ์ใดควรใช้ในการเลือกสร้างอินเด็กซ์
เฉลย: ข — ตารางข้อมูลที่มีจำนวนเรคอร์ดเป็นจำนวนมาก
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดกล่าวถึงวิวได้ไม่ถูกต้อง
เฉลย: จ — "วิวช่วยค้นข้อมูลที่จัดเรียงตามตัวอักษร" ผิด — นั่นคือหน้าที่ของอินเด็กซ์
ข้อ 2. สร้างวิวเลือกเฉพาะบางฟิลด์ มีประโยชน์ด้านใด
เฉลย: ก — การจำกัดการเข้าถึงข้อมูลในตาราง (security)
ข้อ 3. ข้อใดคือวิวข้อมูล (materialized)
เฉลย: ก — วิวที่ดึงข้อมูลจากตารางหลักมาเก็บไว้ในพื้นที่ฐานข้อมูล
ข้อ 4. CREATE VIEW … COUNT(*) … GROUP BY region เป็นวิวแบบใด
เฉลย: ง — วิวกลุ่ม (มี GROUP BY/ฟังก์ชันสรุปผล)
ข้อ 5. ข้อใดกล่าวไม่ถูกต้องสำหรับคำสั่งสร้างวิว view_job
เฉลย: ค — วิวสร้างจากตาราง employee ไม่ใช่ตารางชื่อ view_job (view_job คือชื่อวิว)
ข้อ 6. ข้อใดมีลักษณะการทำงานเหมือนอินเด็กซ์
เฉลย: ค — การค้นหาหนังสือในห้องสมุดด้วยหมายเลขเรียกหนังสือ
ข้อ 7. อินเด็กซ์ถูกสร้างได้ในกรณีใด
เฉลย: จ — ถูกทั้ง ก ข ค (คีย์หลัก, CREATE INDEX, กำหนด UNIQUE)
ข้อ 8. คำสั่งสร้างอินเด็กซ์ผสม
เฉลย: ง — CREATE INDEX supplier_idx ON supplier_product(product_no, supplier_id)
ข้อ 9. คำสั่งอินเด็กซ์ใดช่วยรายงานเรียงวันที่ เก่า→ใหม่
เฉลย: ข — CREATE INDEX … ON order(order_date) — ไม่ใส่ DESC = เก่าไปใหม่
ข้อ 10. เกณฑ์ใดที่ควรเลือกที่จะไม่สร้างอินเด็กซ์
เฉลย: ค — ตารางข้อมูลที่มีการเพิ่ม ลบ แก้ไขข้อมูลบ่อยๆ
สรุปประเด็นที่ออกสอบบ่อย
  • วิว = ตารางเสมือนจาก SELECT ไม่เก็บข้อมูลจริง สะท้อนการเปลี่ยนของตารางหลัก; ประโยชน์ = จำกัดการเข้าถึง + ลดความซับซ้อนคิวรี่
  • ประเภทวิว: ระดับ(แถว/WHERE), ทางดิ่ง(คอลัมน์), กลุ่ม(GROUP BY), วิวข้อมูล(เก็บจริง) vs วิวเสมือน(ไม่เก็บ)
  • อินเด็กซ์ = เหมือนดัชนีหนังสือ; สร้างจาก คีย์หลัก/CREATE INDEX/UNIQUE; ผสม = หลายคอลัมน์
  • DESC = ใหม่→เก่า, ASC(ไม่ใส่) = เก่า→ใหม่
  • ควรสร้าง index: ตารางใหญ่/ค้นบ่อย; ไม่ควร: ตารางแก้ไขบ่อย, ค่าซ้ำมาก, คอลัมน์รูปภาพ
12

การจัดการรายการ การฟื้นสภาพ และการควบคุมภาวะพร้อมกัน

ภาพรวมของหน่วย — ทำให้ข้อมูลถูกต้องเมื่อมีหลายงานทำพร้อมกันหรือเกิดความขัดข้อง แบ่งเป็น 2 ตอน:
• ตอน 12.1 การจัดการรายการ (transaction)
• ตอน 12.2 การฟื้นสภาพ (recovery) และการควบคุมภาวะพร้อมกัน (concurrency control)

ตอนที่ 12.1 การจัดการรายการ (Transaction)

รายการ (transaction) = หน่วยของงานในระดับตรรกะที่ต้องรักษาความถูกต้องของข้อมูล — สำเร็จทั้งชุด หรือถ้าผิดพลาดต้องยกเลิกทั้งชุด (ย้อนกลับสู่สถานะก่อนเริ่ม)

สถานะของรายการ — 5 สถานะ

active (เริ่มทำงาน) → partially committed (ทำคำสั่งสุดท้ายเสร็จ รอคอมมิต) → committed (เสร็จสมบูรณ์ บันทึกแล้ว) · หรือเส้นทางผิดพลาด: active → failed → aborted (ยกเลิก/ย้อนกลับ)

คุณสมบัติ ACID

  • Atomicity: รายการเป็นหน่วยเดียว แบ่งย่อยไม่ได้ — ทำสำเร็จทั้งหมดหรือไม่ทำเลย
  • Consistency: หลังรายการเสร็จ ฐานข้อมูลต้องถูกต้องตามกฎควบคุมเสมอ
  • Isolation (ความเป็นอิสระ) และ Durability (ความคงทน)

ตอนที่ 12.2 การฟื้นสภาพและการควบคุมภาวะพร้อมกัน

การฟื้นสภาพ (Recovery) = ทำให้ข้อมูลย้อนกลับสู่สภาพเดิมที่ถูกต้อง "ก่อน" เกิดปัญหา · วิธี: การย้อนทำ (undo/rollback), การทำซ้ำ (redo), จุดตรวจสอบ (checkpoint) — ใช้ไฟล์ประวัติ (log file) ช่วย; เลือกใช้ได้หลายวิธี

ประเภทความขัดข้อง — 3 ประเภท

ประเภทตัวอย่าง
ของรายการ (transaction)พื้นที่ดิสก์ไม่พอต่อการทำงานของรายการ
ของระบบ (system)ไฟฟ้าดับ, ความผิดพลาดของระบบปฏิบัติการ
ของสื่อบันทึก (media)ตัวควบคุมดิสก์เสีย

ภาวะพร้อมกัน (Concurrency) และการควบคุม

  • ภาวะพร้อมกัน = หลายรายการเรียกใช้ข้อมูลเดียวกันในเวลาเดียวกัน
  • การควบคุม = จัดการการเข้าถึงพร้อมๆ กันไม่ให้ข้อมูลเสียหาย; วิธี: ล็อกกิ้ง, ไทม์แสตมป์, ออปทิมิสทิก
  • ประเภทของล็อก: exclusive lock และ share lock
  • เดดล็อก (deadlock) = ล็อกไม่คลาย (รอทรัพยากรกันวน) ข้อเสียของวิธีล็อกกิ้ง; ตรวจด้วยกราฟ wait-for graph
แนวตอบกิจกรรมระหว่างเรียน
12.1 รายการคืออะไร / สถานะ / คุณสมบัติ ACID
หน่วยงานตรรกะ สำเร็จทั้งชุดหรือยกเลิกทั้งชุด; 5 สถานะ (active→partially committed→committed / failed→aborted); ACID
12.2 การฟื้นสภาพ / ประเภทความขัดข้อง / การควบคุมภาวะพร้อมกัน
recovery (undo/redo/checkpoint + log); ความขัดข้อง 3 แบบ (รายการ/ระบบ/สื่อ); concurrency control (lock/timestamp/optimistic) + deadlock
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ความหมายของรายการ (transaction)
เฉลย: ก — หน่วยงานในระดับตรรกะที่ต้องรักษาความถูกต้องของข้อมูล
ข้อ 2. ข้อใดถูกต้องที่สุดเกี่ยวกับการจัดการรายการ
เฉลย: ก — จัดการหน่วยงานระดับตรรกะที่ทำกับฐานข้อมูล เพื่อรักษาความถูกต้องโดยระบบจัดการ
ข้อ 3. รายการจะอยู่ในสถานะใดสถานะหนึ่ง ยกเว้นข้อใด
เฉลย: จ — "สถานะย้อนการทำงานของรายการอีกครั้ง" ไม่ใช่สถานะของรายการ
ข้อ 4. รายการอ่านเขียนจนถึงคำสั่งสุดท้ายไม่มีปัญหา อยู่สถานะใด
เฉลย: ข — partially committed (เสร็จและรอคอมมิต)
ข้อ 5. ข้อใดไม่ใช่ความหมายของ atomicity
เฉลย: จ — "ฐานข้อมูลต้องถูกต้องสอดคล้องเสมอ" คือ consistency ไม่ใช่ atomicity
ข้อ 6. ข้อใดกล่าวเกี่ยวกับการฟื้นสภาพได้ถูกต้องที่สุด
เฉลย: ก — ทำให้ข้อมูลย้อนกลับสู่สภาพเดิมที่ถูกต้องก่อนเกิดปัญหา
ข้อ 7. ประเภทของความขัดข้องจำแนกได้ 3 ประเภท
เฉลย: ข — ของรายการ / ระบบ / สื่อบันทึกข้อมูล
ข้อ 8. พื้นที่ดิสก์ไม่พอต่อการทำงานของรายการ เป็นความขัดข้องประเภทใด
เฉลย: ค — ความขัดข้องของรายการ (transaction failure)
ข้อ 9. ความหมายของ "ภาวะพร้อมกัน" (concurrency)
เฉลย: ง — หลายรายการเรียกใช้ข้อมูลเดียวกันในเวลาเดียวกัน
ข้อ 10. ความหมายของ "การควบคุมภาวะพร้อมกัน"
เฉลย: ข — จัดการการเข้าถึงข้อมูลพร้อมๆ กัน เพื่อไม่ให้ข้อมูลเสียหาย
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ข้อใดไม่เกี่ยวข้องกับรายการ (transaction)
เฉลย: ง — ข้อที่อธิบาย "การจัดการการเข้าถึง…ตัวจัดการกู้คืน" เป็นการจัดการรายการ ไม่ใช่ตัวรายการ
ข้อ 2. ข้อใดไม่ถูกต้องเกี่ยวกับการจัดการรายการ
เฉลย: ง — ข้อที่นิยามผิดว่าเป็นการจัดการการเข้าถึงข้อมูลด้วยตัวจัดการต่างๆ
ข้อ 3. ลำดับสถานะของรายการที่ถูกต้อง
เฉลย: ก — active → partially committed → committed
ข้อ 4. สถานะที่การกระทำเสร็จและรอคอมมิต
เฉลย: จ — partially committed (ทำคำสั่งสุดท้ายโดยไม่มีปัญหา)
ข้อ 5. ความหมายของ atomicity
เฉลย: ก — รายการหลายคำสั่งถูกมองเป็นหน่วยเดียว แบ่งย่อยไม่ได้
ข้อ 6. ข้อใดไม่ถูกต้องเกี่ยวกับการฟื้นสภาพ
เฉลย: ง — "เลือกใช้วิธีใดวิธีหนึ่งเท่านั้น" ผิด — ใช้ได้หลายวิธี (undo/redo/checkpoint)
ข้อ 7. ข้อใดไม่ใช่ประเภทของความขัดข้อง
เฉลย: จ — "ความขัดข้องของตลาดไอเท็ม" ไม่ใช่ประเภทความขัดข้อง
ข้อ 8. ตัวอย่างความขัดข้องของรายการ (transaction failure)
เฉลย: ข — พื้นที่ดิสก์ไม่เพียงพอต่อการทำงานของรายการ
ข้อ 9. ข้อใดไม่เกี่ยวข้องกับ "ภาวะพร้อมกัน"
เฉลย: จ — ข้อที่นิยามภาวะพร้อมกันคลาดเคลื่อน
ข้อ 10. ข้อใดไม่เกี่ยวข้องกับ "การควบคุมภาวะพร้อมกัน"
เฉลย: ข — ประเภทล็อกที่ถูกต้องคือ exclusive lock และ share lock
สรุปประเด็นที่ออกสอบบ่อย
  • รายการ (transaction) = หน่วยงานตรรกะ ทำสำเร็จทั้งชุดหรือยกเลิกทั้งชุด; 5 สถานะ active→partially committed→committed / failed→aborted
  • ACID: Atomicity (ทั้งหมดหรือไม่ทำ), Consistency (ถูกต้องเสมอ), Isolation, Durability
  • ฟื้นสภาพ (recovery) = ย้อนสู่สภาพถูกต้องก่อนปัญหา; วิธี undo/redo/checkpoint + log file
  • ความขัดข้อง 3 ประเภท: รายการ (ดิสก์ไม่พอ) / ระบบ (ไฟดับ, OS) / สื่อบันทึก (ดิสก์เสีย)
  • ภาวะพร้อมกัน ควบคุมด้วย locking/timestamp/optimistic; ล็อก = exclusive/share; เดดล็อกตรวจด้วย wait-for graph
13

การบริหารระบบจัดการฐานข้อมูล

ภาพรวมของหน่วย — คน-หน้าที่ในการดูแลฐานข้อมูล และการทำให้ฐานข้อมูลทำงานเร็ว/มีประสิทธิภาพ แบ่งเป็น 2 ตอน:
• ตอน 13.1 บุคลากร + หน้าที่ของผู้บริหารฐานข้อมูล (DBA)
• ตอน 13.2 ประสิทธิภาพและการปรับแต่งประสิทธิภาพระบบ

ตอนที่ 13.1 บุคลากร + หน้าที่ผู้บริหารฐานข้อมูล (DBA)

บุคลากรที่เกี่ยวข้อง (ใครทำอะไร)

  • นักวิเคราะห์ระบบ: รวบรวมความต้องการของผู้ใช้ แล้วทำเป็นข้อกำหนดการทำงาน
  • นักออกแบบฐานข้อมูล: พัฒนา "วิว" ให้ตรงกับความต้องการและกระบวนการทำงานของผู้ใช้
  • ผู้บริหารข้อมูล (DA) / ผู้บริหารฐานข้อมูล (DBA): ดูแลนโยบายและการบริหารฐานข้อมูล

หน้าที่ของ DBA แบ่งเป็น 3 ด้าน

  • ด้านบริหาร: รักษาความถูกต้องของข้อมูล, กำหนดมาตรฐาน/นโยบาย (เช่น บังคับให้มี username/password = นโยบาย)
  • ด้านสนับสนุนผู้ใช้: รวบรวมความต้องการผู้ใช้, ฝึกอบรมการใช้งาน
  • ด้านบำรุงรักษา: อัพเกรดซอฟต์แวร์, บริหารอุปกรณ์จัดเก็บ, ติดตั้งเครื่องมือ (⚠ การควบคุมภาวะพร้อมกันไม่ใช่งานบำรุงรักษา)

มาตรการสำรอง/กู้คืน: สำรองข้อมูล+โปรแกรมเป็นระยะ, ทดสอบแผนกู้คืน/แผนฉุกเฉินสม่ำเสมอ, มีอุปกรณ์สำรองที่สะดวกและปลอดภัย

ตอนที่ 13.2 ประสิทธิภาพและการปรับแต่ง

ปัจจัยที่มีผลต่อประสิทธิภาพ query = ฮาร์ดแวร์ (CPU, หน่วยความจำหลัก, ฮาร์ดดิสก์, ระบบเครือข่าย) — สิ่งที่ "ไม่มีผล" โดยตรงคือตัวบุคคล (ผู้บริหารฐานข้อมูล/ผู้ใช้งาน)

ขั้นตอนการประมวลผลแบบสอบถาม: แยกย่อยและวิเคราะห์ (parsing) → แปลง SQL ให้มีประสิทธิภาพ + สร้างแผนการเข้าถึง (optimization) → ประมวลผล/ดำเนินการ (execution, มีการล็อกข้อมูล)

การปรับแต่ง (Tuning)

  • เขียน SQL ที่ดี: หลีกเลี่ยงเงื่อนไขที่เป็นนิพจน์/ฟังก์ชัน, AND → เขียนเงื่อนไขที่มักเป็นเท็จก่อน, OR → เขียนเงื่อนไขที่มักเป็นจริงก่อน, หลีกเลี่ยง NOT
  • ปรับแต่งการจัดเก็บกายภาพ: สร้างอินเด็กซ์, กำหนดขนาดแคช, ดีนอร์มัลไลซ์ (denormalize), สร้างสถิติฐานข้อมูล
  • อินเด็กซ์: ไม่ช่วยการ “สร้างตาราง”; ไม่ควรสร้างบนคอลัมน์ในตารางขนาดเล็ก; แคชจัดเรียง (sort cache) ใช้กับ GROUP BY
แนวตอบกิจกรรมระหว่างเรียน
13.1 บุคลากรที่เกี่ยวข้อง / หน้าที่ของ DBA
นักวิเคราะห์ระบบ(เก็บความต้องการ), นักออกแบบ(สร้างวิว); DBA แบ่งหน้าที่ บริหาร/เทคนิค/สนับสนุนผู้ใช้ + สำรอง-กู้คืน
13.2 ประสิทธิภาพ / การปรับแต่ง
ปัจจัย = ฮาร์ดแวร์; query processing = parse→optimize→execute; ปรับแต่งด้วย index/แคช/ดีนอร์มัลไลซ์/สถิติ + เขียน SQL ให้ดี
แบบประเมินผลตนเองก่อนเรียน (พร้อมเฉลย)
ข้อ 1. ใครพัฒนาวิวให้ตรงกับความต้องการและกระบวนการของผู้ใช้
เฉลย: ค — นักออกแบบฐานข้อมูล
ข้อ 2. ข้อใดไม่ใช่ปัจจัยทางเทคนิคในการนำ DBMS มาใช้
เฉลย: ค — ความสามารถในการโยกย้ายไปใช้แพลตฟอร์มอื่น (ตามเฉลย)
ข้อ 3. หน้าที่ทางการบริหารของผู้ดูแลฐานข้อมูล
เฉลย: ก — การรักษาความถูกต้องของข้อมูล
ข้อ 4. หน้าที่ด้านการสนับสนุนผู้ใช้ของ DBA
เฉลย: ก — การรวบรวมความต้องการของผู้ใช้
ข้อ 5. ข้อใดไม่ใช่มาตรการสำรองและกู้คืนข้อมูล
เฉลย: จ — ข้อที่ไม่เกี่ยวกับการสำรอง/กู้คืนโดยตรง
ข้อ 6. ข้อใดไม่ใช่ปัจจัยที่ส่งผลต่อประสิทธิภาพฐานข้อมูล
เฉลย: ข — ผู้ใช้งาน (ไม่ใช่ปัจจัยด้านประสิทธิภาพ)
ข้อ 7. ข้อใดไม่มีผลต่อประสิทธิภาพการประมวลผลแบบสอบถาม
เฉลย: จ — ผู้บริหารฐานข้อมูล (ตัวบุคคล ไม่ใช่ทรัพยากรประมวลผล)
ข้อ 8. ข้อใดกล่าวถึงสถิติฐานข้อมูลไม่ถูกต้อง
เฉลย: ง — "ทุกระบบสร้างสถิติให้อัตโนมัติ" ผิด — ไม่ใช่ทุกระบบ
ข้อ 9. ข้อใดไม่ควรปฏิบัติในการเขียนคำสั่ง SQL
เฉลย: ข — หลีกเลี่ยง = แล้วใช้ != แทน เป็นแนวทางที่ผิด
ข้อ 10. กำหนดพารามิเตอร์ใดให้พอสำหรับ GROUP BY
เฉลย: ค — แคชจัดเรียง (sort cache)
แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ใครรวบรวมความต้องการของผู้ใช้แล้วทำเป็นข้อกำหนดการทำงาน
เฉลย: ง — นักวิเคราะห์ระบบ
ข้อ 2. ข้อใดเป็นปัจจัยด้านการออกแบบฐานข้อมูลทางกายภาพ
เฉลย: ก — อัตราการทำงานของทรานแซกชัน
ข้อ 3. ข้อใดเกี่ยวกับหน้าที่ทางเทคนิคของผู้ดูแลฐานข้อมูล
เฉลย: ก — ตามเฉลย (ดูตัวเลือกในเล่ม)
ข้อ 4. ข้อใดไม่ใช่หน้าที่ด้านการบำรุงรักษาระบบของ DBA
เฉลย: จ — การควบคุมภาวะพร้อมกัน (เป็นงานปฏิบัติการ ไม่ใช่บำรุงรักษา)
ข้อ 5. บังคับให้ผู้ใช้ทุกคนมีชื่อผู้ใช้และรหัสผ่าน เป็นการดำเนินการตามข้อใด
เฉลย: ก — นโยบาย (policy)
ข้อ 6. ข้อใดกล่าวเกี่ยวกับการปรับแต่งประสิทธิภาพได้ถูกต้อง
เฉลย: ค — ลดความซับซ้อน/ตอบสนองคำขอของไคลเอนต์ให้เร็วที่สุด
ข้อ 7. ข้อใดเป็นงานในขั้น execution ของแบบสอบถาม
เฉลย: ค — การล็อกข้อมูล (ทำในขั้นดำเนินการ)
ข้อ 8. อินเด็กซ์ไม่ได้ช่วยเพิ่มประสิทธิภาพในการดำเนินการใด
เฉลย: จ — การสร้างตาราง
ข้อ 9. คอลัมน์ใดไม่ควรสร้างอินเด็กซ์
เฉลย: จ — คอลัมน์ในตารางขนาดเล็ก
ข้อ 10. ข้อใดเป็นการปรับแต่งการจัดเก็บข้อมูลทางกายภาพ
เฉลย: ค — การดีนอร์มัลไลซ์ (denormalize)
สรุปประเด็นที่ออกสอบบ่อย
  • ใครทำอะไร: นักวิเคราะห์ระบบ=เก็บความต้องการ, นักออกแบบ=สร้างวิว, DBA=บริหารฐานข้อมูล
  • หน้าที่ DBA 3 ด้าน: บริหาร / เทคนิค / สนับสนุนผู้ใช้; username+password = นโยบาย
  • ปัจจัยประสิทธิภาพ query = ฮาร์ดแวร์ (CPU/RAM/ดิสก์/เครือข่าย) ไม่ใช่ตัวบุคคล
  • query processing: parse → optimize(สร้างแผนเข้าถึง) → execute(ล็อกข้อมูล)
  • ปรับแต่ง: index/แคช/ดีนอร์มัลไลซ์/สถิติ; index ไม่ช่วยสร้างตาราง และไม่ควรสร้างบนตารางเล็ก
14

การประยุกต์ใช้งานฐานข้อมูล

ภาพรวมของหน่วย — เทคโนโลยีฐานข้อมูลสมัยใหม่และการประยุกต์ แบ่งเป็น 3 ตอน:
• ตอน 14.1 เทคโนโลยี: กระจาย/ขนาน, คลังข้อมูล, ข้อมูลขนาดใหญ่
• ตอน 14.2 ฐานข้อมูลเชิงวัตถุ (OO) และโนเอสคิวแอล (NoSQL)
• ตอน 14.3 ฐานข้อมูลผ่านเว็บและ XML

ตอนที่ 14.1 เทคโนโลยีการใช้งานฐานข้อมูล

ฐานข้อมูลแบบกระจาย (Distributed)

  • ข้อมูลกระจายเก็บหลายเครื่องที่เชื่อมด้วยเครือข่าย
  • แบบระบบเดียวกัน (homogeneous): ทุกเครื่องใช้ DBMS ชนิดเดียวกัน + สคีมาร่วมกัน
  • แบบหลากหลาย (heterogeneous): DBMS ต่างกัน + สคีมาต่างกันในแต่ละไซต์

ฐานข้อมูลแบบขนาน (Parallel): เพิ่ม throughput, ลดเวลาตอบสนอง, รองรับการขยายระบบ, เพิ่มความน่าเชื่อถือ — (การแบ่งปันข้อมูล ไม่ใช่จุดเด่นของแบบขนาน)

คลังข้อมูล (Data Warehouse): ETL (Extract-Transform-Load) = นำข้อมูลจากหลายแหล่งไปไว้ในคลังข้อมูล · OLAP = เครื่องมือวิเคราะห์ข้อมูลหลายมิติ ดริลล์ดาวน์/หมุนแกนลูกบาศก์

ข้อมูลขนาดใหญ่ (Big Data): เช่น แนะนำโฆษณา/สินค้า (Google/Facebook/Amazon), พยากรณ์แผ่นดินไหว — ไม่ใช่ big data: ค้นรายชื่อหนังสือในเว็บห้องสมุด, ตอบแบบสอบถามออนไลน์, ทะเบียนเรียน (เป็นคิวรี่ทั่วไป)

ตอนที่ 14.2 ฐานข้อมูลเชิงวัตถุและ NoSQL

  • คลาส (เชิงวัตถุ) = แอตทริบิวต์ (attribute = ข้อมูล เช่น name, studentAddress) + พฤติกรรม/โอเปอเรชัน (operation/method = การกระทำ เช่น addCourse, deleteCourse)
  • ORM (Object-Relational Mapping): เครื่องมือที่ทำให้โปรแกรมเชิงวัตถุใช้วัตถุในฐานข้อมูลได้
  • NoSQL 4 ประเภท: คีย์-ค่า (key-value), เอกสาร (document — เช่น MongoDB), คอลัมน์ (column — เช่น Apache Cassandra), กราฟ (graph)

ตอนที่ 14.3 ฐานข้อมูลผ่านเว็บและ XML

  • การทำงานของเว็บไซต์: เว็บเพจสถิต (static) และ เว็บเพจพลวัต (dynamic)
  • ส่วนขยายฝั่งเซิร์ฟเวอร์ (server-side extension): ซอฟต์แวร์ที่จัดการคำร้องขอระหว่างฐานข้อมูลและเว็บเซิร์ฟเวอร์

การรักษาความปลอดภัยของระบบฐานข้อมูล

ภาพรวมของหน่วย (หน่วยสุดท้าย) — การป้องกันฐานข้อมูลจากภัยคุกคาม แบ่งเป็น 3 ตอน:
• ตอน 15.1 หลักการพื้นฐาน: จำแนกชั้นความลับ + ภัยคุกคามและการป้องกัน
• ตอน 15.2 การควบคุมการเข้าใช้งาน (สิทธิ์ GRANT/REVOKE)
• ตอน 15.3 ความปลอดภัยกับการประยุกต์ใช้งาน (ฐานข้อมูลเชิงสถิติ, inference)

ตอนที่ 15.1 หลักการพื้นฐานการรักษาความปลอดภัย

การจำแนกชั้นความลับข้อมูล: 4 ระดับ — ลับที่สุด, ลับมาก, ลับ, เปิดเผย · องค์ประกอบที่ใช้พิจารณา: ความสำคัญของเนื้อหา, แหล่งที่มา, วิธีนำไปใช้ประโยชน์, บุคคลที่ควรรับทราบ

ภัยคุกคาม = การสูญเสีย CIA

  • Confidentiality (ความลับ): การเปิดเผยข้อมูลโดยไม่ได้รับอนุมัติ = สูญเสียความลับ
  • Integrity (ความคงสภาพ) และ Availability (ความพร้อมใช้งาน)

มาตรการป้องกัน: การควบคุมการเข้าถึง (access control), การควบคุมอนุมาน (inference control), การควบคุมการไหลของข้อมูล (flow control), การเข้ารหัส (encryption) — (⚠ server control ไม่ใช่มาตรการ) · encryption ป้องกันการสูญเสียความลับและความคงสภาพ

ศัพท์: ช่องทางแอบแฝง (covert channel) = ช่องทางไหลข้อมูลที่ไม่เป็นไปตามนโยบายความปลอดภัย · Bell-LaPadula: X อ่าน Y ได้ ถ้า class(X) ≥ class(Y); X เขียน Y ได้ ถ้า class(X) ≤ class(Y)

ตอนที่ 15.2 การควบคุมการเข้าใช้งาน

ระดับการให้สิทธิ์ — 2 ระดับ: ระดับบัญชีผู้ใช้ (account level — DBA ระบุสิทธิ์แต่ละบัญชี อิสระจากความสัมพันธ์) · ระดับความสัมพันธ์/ตาราง (relation/table level — สิทธิ์ผูกกับตารางแต่ละตาราง)

GRANT UPDATE ON T1 TO U2;        -- U2 อัปเดตตาราง T1 ได้
REVOKE SELECT ON STUDENT FROM B;  -- ดึงสิทธิ์ SELECT คืนจาก B

การควบคุมการเข้าถึงแบบบุคคล: ให้สิทธิ์ระดับไฟล์ / เรคอร์ด / แต่ละฟิลด์

ตอนที่ 15.3 ความปลอดภัยกับการประยุกต์ใช้งาน

  • ฐานข้อมูลเชิงสถิติ: เก็บเฉพาะข้อมูลสถิติ หรือใช้ทั่วไปแต่ให้ผลทางสถิติ
  • ภัย inference (การอนุมาน): ใช้คิวรี่หลายครั้งอนุมานข้อมูลส่วนบุคคล
  • ป้องกัน inference: การจำกัดคิวรี่ (query restriction) + เพอร์เทอร์เบชัน (perturbation = ปรับเปลี่ยนข้อมูลให้ได้ค่าประมาณแทนค่าจริง)
  • การไหลของข้อมูล: การไหลโดยตรง (explicit เช่น @x = @y + 9) vs การไหลโดยอ้อม (implicit)

เฉลย: ง — ข้อ ก. และ ข. ถูก (เก็บเฉพาะสถิติ / ใช้ทั่วไปแต่ให้ผลสถิติ)

แบบประเมินผลตนเองหลังเรียน (พร้อมเฉลย)
ข้อ 1. ความสำคัญของเนื้อหาและแหล่งที่มา ใช้พิจารณากำหนดข้อใด
